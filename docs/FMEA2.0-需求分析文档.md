## 1. 项目概述

### 1.1 项目背景

FMEA（Failure Mode and Effects Analysis，失效模式与影响分析）2.0 系统旨在为企业产品研发过程中的 DRBFM（Design Review Based on Failure Mode，基于失效模式的设计评审）提供 IT+AI 一体化支撑平台。系统覆盖从变更点识别、风险评估、DRBFM分析、基线输出、评审审批到基线落地跟踪与入库的全生命周期管理。

### 1.2 项目目标

1. **流程数字化**：将 DRBFM 从线下手工操作转为线上系统化流程，实现变更点清单导入、风险评估、结构分析、功能分析、失效分析、评审审批的全流程在线管理。
2. **AI辅助分析**：通过 AI 模型辅助完成五维风险评估、功能分析、失效模式识别、失效原因分析、预防/探测措施推荐、SOD评分等工作，提升分析效率与质量。
3. **知识沉淀与复用**：建立基线库、失效库、措施库、功能库等知识资产，实现跨项目知识复用。
4. **过程可追溯**：全流程数据留痕，支持版本迭代、评审记录、审批记录的追溯。

### 1.3 系统定位

FMEA 2.0 是面向研发团队的质量风险管理平台，核心用户为 TSE（测试与验证工程师）、PSE（产品系统工程师）及各专业域工程师。

### 1.4 系统约束

| 约束项 | 确认结果 |
|--------|----------|
| 终端支持 | 仅支持PC端，不支持移动端 |
| 用户规模 | 约100人（并发用户数） |
| 语言支持 | 一期仅支持中文，后续再考虑国际化 |
| PMS集成方式 | 仅做数据推送/拉取，不深度集成 |

## 2. 业务流程全景

### 2.1 主流程

FMEA 2.0 的核心业务流程如下：

```
DRBFM触发评估 → DRBFM分析 → 基线输出 → 评审 → 审批 → 基线落地跟踪 → 入库
```

### 2.2 流程执行顺序

系统级和部件级 DRBFM 有严格的先后顺序：

1. **系统级 DRBFM**：在项目 TR1~TR3 之间进行，由项目 TSE 主导
2. **部件级 DRBFM**：在项目 TR3~TR4 之间进行，是对系统级的细化，由 TSE 指派协作人

**关键约束**：
- 一个项目只能有一个系统级 DRBFM 分析任务
- 部件级是对系统级的细化，必须先完成系统级
- 一期仅实现系统级和部件级各一次触发，暂不实现 TR4a/TR5/TR6 的迭代

### 2.3 TR节点触发机制

- TR节点之间的迭代更新由 **PMS系统推送触发**，非人工触发
- 迭代更新采用 **方案1**：1.0版本后每个TR节点4周前，推送所有变更点信息，进行1次评审
- 迭代更新时，之前版本的分析内容采用 **增量更新** 方式处理
- 迭代更新需要重新走完整的评审审批流程

### 2.4 版本迭代与TR节点对应

| | TR1 | TR3 | TR4 | TR4a | TR5 | TR6 |
|------|-----|-----|-----|------|-----|-----|
| 系统FMEA | 触发 | 1.0版本 | 迭代2.0 | 迭代3.0 | 迭代4.0 | 迭代5.0 |
| 部件级FMEA | — | 触发 | 1.0版本 | 迭代2.0 | 迭代3.0 | 迭代4.0 |

> 注：一期仅实现系统级（TR1触发→TR3发布1.0）和部件级（TR3触发→TR4发布1.0）各一次。

## 4. DRBFM触发评估

### 4.1 系统级DRBFM触发评估

#### 4.1.1 流程描述

1. **TSE导入变更点清单**：变更清单包含三部分信息——结构组成（Level、ID、系统/部件名称）、开发情况（开发类别、I/M/N）、变更点描述（变更点）。
2. **TSE导入质量策划表**：从质量策划表中提取"风险小类+风险策划指南"字段组合，根据变更点清单中的部件名称进行模糊匹配，更新变更点描述。
3. **自动匹配**：点击"自动匹配"按钮，系统读取线上表单内容，匹配质量策划内容进行更新。
4. **TSE提交审核**：TSE确认变更清单后提交，任务转PSE审核。
5. **PSE审核**：
   - 审核通过 → 任务自动转回TSE，TSE提交历史问题（需与系统级评估任务关联），生成DRBFM评估表。
   - 审核不通过 → 通知TSE（飞书/邮件），TSE重新提交。
6. **五维风险评估**：对LEVEL=0的项目进行五维评估（技术新颖性、影响范围、失效严重度、变更复杂度、历史问题），每维度填写"是/否/空白"。
7. **风险综合评分**：用户填完五维信息后点击评分按钮，系统计算风险综合评分。
8. **PSE和TSE复核**：确认评分评估表完整性后，生成DRBFM分析任务。
9. **撤回机制**：评估表提交后允许撤回修改。

#### 4.1.2 导入顺序规则

| 场景 | 处理逻辑 |
|------|----------|
| 先导入变更清单，再导入质量策划 | 变更清单直接更新线上表格；质量策划根据线上变更清单匹配更新 |
| 先导入质量策划，再导入变更清单 | 质量策划导入时线上变更清单为空，不触发匹配；变更清单导入时读取质量策划内容匹配更新 |
| 点击自动匹配 | 读取线上表单内容，匹配质量策划内容进行更新 |

#### 4.1.3 变更点清单导入

- **导入格式**：Excel模板，有标准模板
- **导入字段**：LEVEL、ID、系统/部件名称、开发类型、I/M/N选项、变更点
- **结构层级**：支持无限层级
- **开发类别**：暂时手动输入，无固定选项列表

#### 4.1.4 质量策划表导入

- **导入格式**：读取Excel文件，判断sheet内容，转markdown后由AI处理
- **模糊匹配**：暂时确认不了精度要求，先支持导入，匹配功能暂不实现

#### 4.1.5 风险综合评分规则

- **计算公式**：风险综合评分 = 技术新颖性×15% + 影响范围×30% + 失效严重度×30% + 变更复杂度×20% + 历史问题×5%
- **五维评估选项映射**：每个维度有多个选项（高有3个选项、中有2个选项、低有2个选项），如果"高"的选项中有"是"，则取"高"对应的分数；"高"没有时，看"中"的选项是否有"是"，有则取"中"对应的分数；以此类推。
- **结论判定**：
  - ≥8分（H）：必须进行DRBFM分析
  - 5-8分（M）：建议进行DRBFM分析
  - ＜5分（L）：可以不进行DRBFM分析
- **DRBFM建议**：不涉及AI生成，用户可选择两种模式之一（分数映射预定义理由 / AI根据评分动态生成），由用户在页面上选择。

#### 4.1.6 历史问题

- **数据来源**：手动导入
- **系统考虑建立历史问题库**
- **历史问题字段**：故障单号（如有请补充）、问题编号、问题类型、涉及领域、产品型号、问题描述、详细说明、解决方案
- 历史问题需与系统级评估任务关联
- 部件级评估任务需读取系统级评估任务的历史问题

#### 4.1.7 数据模型要求

- 记录提交次数
- 评估表提交后允许撤回修改

### 4.2 部件级DRBFM触发评估

#### 4.2.1 与系统级的差异

1. TSE可指派多名协作人（单方面指派），各协作人可编辑各自负责的变更点。
2. 变更点清单和质量策划由TSE统一上传，TSE指定协作人后给予编辑权限。
3. 协作人导入的变更点清单可能不完整，需要合并算法：**根据ID合并变更点内容**。
4. DRBFM评估表合并规则：保留系统级（LEVEL=0）的评估内容，更新部件级评估内容，按行ID匹配——已有评估内容的保留，否则新增（双路排序算法）。
5. 增加协作人签核步骤：所有被指派的协作人签核后，TSE才能发起提交，然后PSE审核。

## 5. DRBFM分析

DRBFM分析任务由评估任务完成后根据评估结论创建，初始化多个子任务，各子任务生成关键交付物。

### 5.1 结构分析

#### 5.1.1 功能描述

1. **交付物**：结构框图（创建飞书文档，在文档中创建画板节点），记录文档及画板节点关键字，交付物类型为结构框图。
2. **飞书集成**：PMS系统已有集成飞书文档及飞书画板的相关接口，可复用。
3. **结构框图生成方式**：
   - **方式A - 一键导入**：如果项目之前绘制过结构框图，可一键导入（结构框图在进入下一环节时会生成节点及边的关系，可用来重新生成图）。
   - **方式B - 从系统级读取**：部件级分析任务可读取当前项目系统级分析任务的最新结构框图资料，提取关联部件的结构。
   - **方式C - 从评估表生成**：从头开始画时，从评估分析表读取节点组成成分（按层级关系），解读为树结构构造节点信息。建议以 Mermaid 语法生成初图，以画板格式保存。
4. **用户补充**：用户可补充接口表格（起点、终点、功能、功能类型）。
5. **风险标记**：框图节点根据评估分析表中的风险评分标记颜色（红=H、黄=M、灰=L）。
6. **导出**：支持导出为图片/PDF。

#### 5.1.2 框图转换规则

- 从DRBFM触发评估表转换成系统框图内部组成（IT工具实现）
- 连接关系解析：子部件间有连接关系用直线连接；单向显示箭头；双向或无流向不显示箭头

#### 5.1.3 接口模型

依据4大类接口进行线型颜色标记：

| 接口类型 | 线型颜色 |
|----------|----------|
| 物理 | 黑色 |
| 物质 | 蓝色 |
| 能量 | 绿色 |
| 信号 | 紫色 |

### 5.2 变更点关联功能分析

#### 5.2.1 功能描述

1. **交付物**：功能矩阵图。
2. **功能矩阵生成**：读取结构图中的节点信息（到器件一级）和边信息，将器件（节点）作为横轴，功能（边）作为纵轴，形成功能矩阵图，有关联的单元标记"Y"。有H/M风险的标记颜色。
3. **功能来源**：不仅从结构图的边信息提取，**也需要AI从附件中提取**功能。
4. **功能排列顺序**：按功能的丰度（关联的接口数量）排列。
5. **附件导入**：
   - 历史问题（关联到系统级评估任务）
   - 技术要求：暂无固定数据格式，可能是Word文档，需AI处理
   - 需求分析文档（可多份）
6. **AI补充**：上传附件后点击AI补充按钮，按规范生成相关功能，填充到纵轴。
7. **功能库**：支持专业域配置历史功能库。

#### 5.2.2 功能描述规范

- 功能描述：动词+名称（量词或状语）
- 技术要求描述：XX性能+量词
- 功能描述必须清楚，功能之间无重叠冲突
- 一个系统元素可以有多个功能

#### 5.2.3 功能矩阵到DRBFM表单的转换

输入：功能矩阵；输出：DRBRM表单/DFMEA表单"项目/功能"列；横轴与纵轴组合成DRBFM表单"项目/功能"列。

### 5.3 失效分析、预防与探测控制、风险评级

> 失效分析、预防与探测控制、风险评级**合并为一个步骤页面**。

#### 5.3.1 失效模式

失效模式基于功能的偏差，分为7种类型：

| 类型 | 描述 |
|------|------|
| 异常 | 功能执行错误 |
| 相逆 | 功能反向执行 |
| 伴随 | 伴随不需要功能 |
| 过量 | 输出超出范围 |
| 减量 | 输出不足 |
| 部分 | 功能部分缺失 |
| 空白 | 功能完全缺失 |

AI生成失效模式时，需同时关注变更点（关联关键结构/接口）可能带来的典型失效模式，不同失效模式分行列出。

#### 5.3.2 失效原因

失效原因从6个维度展开：

| 维度 | 说明 |
|------|------|
| 固有设计缺陷 | 设计本身存在的问题 |
| 制造/装配问题 | 生产过程中引入的问题 |
| 应力诱发失效 | 外部应力导致的失效 |
| 耗损与老化 | 长期使用导致的退化 |
| 交互性/系统性失效 | 系统间交互导致的问题 |
| 综合应力作用 | 多种应力综合作用 |

**关键规则**：
- 失效原因需结合变更点具象化
- 控制方法成熟的失效原因不用展开
- 其他系统失效且自身无法强化/检测的不用分析
- 以最大可能预防失效为原则

#### 5.3.3 失效影响

- 逐层识别上级/整机/客户级影响，分别列出
- 需具体描述影响什么功能、什么性能
- 多个影响列在一行中，无需分行
- 不同失效影响如相同，需保证描述一致

#### 5.3.4 预防与探测控制

**预防措施**：
- 预防类型优先原则：设计固有可靠性 > 过程控制 > 状态监控
- 预防措施与性能要求相关
- 描述必须清楚、完整，必要时引用参考文献/标准
- 将设计的安全边际作为预防措施考虑
- 针对某个失效原因的所有预防措施写在一行

**探测措施**：
- 探测措施三要素：探测阶段+探测对象+探测方法
- 能够探测失效原因、失效模式的探测措施都应填写（优先考虑探测失效原因）
- 如单个探测措施有效性不够，可结合不同对象/时机/方法的探测措施
- 探测度反映探测措施逼近真实失效机理的程度

#### 5.3.5 风险评级（SOD评价）

- **严重度（S）**：评价功能失效模式导致的最严重影响的度量
- **发生度（O）**：衡量预防措施有效性的指标
  - 模式1（适合解决方案或软件）和模式2（适合硬件/结构件）采用**不同的评分规则**
- **探测度（D）**：表示对失效原因或失效模式探测有效性的估计量度
- SOD彼此独立评估
- **AI给出建议值，用户可自行修改**

#### 5.3.6 AP（Action Priority）判定

根据SOD组合查AP参考表（完整100行），判定优先级为H/M/L。AP参考表为固定标准，暂不允许修改。

#### 5.3.7 探测度评价标准适配关键点

1. **探测方法**：边界试验 < 常规条件试验；综合应力 < 单一应力；监控参数变化 < 判定性能是否满足
2. **探测时机**：早期探测优于后期探测
3. **探测对象**：模块/部品 < 系统/整机；失效原因 < 失效模式
4. **标准定期验证与更新**：收集逃逸问题，反向审视评分和措施有效性

#### 5.3.8 AI生成流程

1. 点击AI生成，读取功能矩阵中变更相关的功能
2. 自动生成失效模式、失效原因、预防措施等，存入临时表
3. 用户在页面上选择后，生成到在线表中，临时表标记"已采纳"
4. 生成后进行SOD评分（AI给出建议值，用户可修改）
5. **支持"重新生成"按钮**，AI生成结果可多次重新生成
6. **临时表管理**：没有有效期，每个分析任务对应的只有一份，每次AI生成时删除前一次生成的内容

## 6. 基线输出与落地跟踪

### 6.1 基线输出

#### 6.1.1 输出模式

- **模式一**：根据失效分析表的严重度×发生度列出预防措施清单、探测措施和优化措施
- **模式二**：直接列出所有预防措施、探测措施、优化措施
- **用户可在页面上切换两种模式**

#### 6.1.2 措施定义

预防措施、探测措施和优化措施都是对失效模式和失效原因的修复措施，三者并列关系。

#### 6.1.3 基线清单管理

1. 工程师判断措施重复，进行去重（删除）
2. 措施标记来源："AI生成/历史"，标记类型："新增/优化"
3. 每个措施需判断是否落地（默认"是"），改为"否"则关闭落地
4. 选择"是"的措施需指定落地负责人和落地时间
5. 提交时校验：所有"是否落地"为"是"的措施，落地负责人和落地时间不能为空，时间不能为过去时间

### 6.2 基线落地跟踪

#### 6.2.1 流程描述

1. 基线跟踪任务展示措施内容
2. 负责人选择落地情况：完成/试验失败/无法落地/延期
3. 提交个人主管审核
4. 主管审核结果：

| 落地情况 | 审核通过后动作 |
|----------|----------------|
| 完成 | 回写基线落地表；写入系统措施库和失效库 |
| 试验失败/无法落地 | **需要重新分析** |
| 延期 | 重新指定负责人/落地时间 |

- 审核不通过：通知负责人重新处理，**无重试次数限制**

#### 6.2.2 SLA要求

基线落地跟踪有时间周期的SLA要求（具体SLA指标待确认）。

#### 6.2.3 措施库与失效库关系

- 一个失效库中的记录对应措施库中的多条记录（预防措施、探测措施、优化措施）
- 措施库具体字段结构暂不明确，待后续细化

## 7. 评审与审批

### 7.1 评审

#### 7.1.1 评审流程

1. **发起评审**：负责人指定评审角色与评审人（项目内动态配置），设置评审日期，发起评审。
2. **评审展示**：评审页面展示DRBFM分析内容及基线清单（无论是否落地的措施都列出）。
3. **评审通知**：
   - 评审日期**提前一周**通知一次，**提前一天**再通知一次
   - 到期未评审则自动不通过
4. **评审结论判定**：
   - 判断项目是否属于1级项目（**从PMS系统获取**）
   - 计算评审人意见率是否达到2/3
   - 未达到2/3：负责人组织线下会议，上传会议纪要（文本），**会议纪要需关联到评审记录**
   - 达到2/3：进行意见闭环（意见提出→负责人答复→提出人闭环）
5. **评审意见闭环**：无时间限制，但**主页需要有入口**方便跟踪
6. **评审纪要**：意见全部闭环后，AI生成评审纪要（文本）。
7. **确认发起审批**：负责人确认后发起审批。

#### 7.1.2 评审角色

必选：专业组长、PSE、TSE、硬件质量经理、DFX SE  
可选：维护代表、制造代表、软件/固件工程师、射频/天线工程师、结构/热设计工程师、采购/供应商质量代表

**评审人不可委托他人代为评审**（暂不允许）。

### 7.2 审批

#### 7.2.1 审批层级

| 决策层级 | 审批者 | 变更类型 |
|----------|--------|----------|
| Level 1 | 专业组长 | 初级变更：不影响接口的软件Bug修复；不涉及软件变更的B配套器件导入；文档勘误；不改变意图的布线优化 |
| Level 2 | 部门经理 | 中度变更：影响单一模块性能的硬件设计优化；软件功能模块版本升级；涉及软件变更的B配套器件导入；涉及重新认证的B配套器件导入 |
| Level 3 | 研发总监 | 重大变更：架构性与范围性变更；资源性与流程性变更；显著影响成本与产品路标变更 |

#### 7.2.2 审批流程

1. 页面展示DRBFM分析内容、基线清单、会议纪要
2. 负责人选择审批人（界面提示对应层级）
3. 审批人收到通知，查看内容后给出审批意见和审批结论
4. 审批不通过：通知发起人，发起人查看审批意见历史，可再次发起审批（**无次数限制**）
5. 审批通过：生成基线跟踪任务

**审批为简单审批流，不支持多级会签**（不需要同时Level 1和Level 2审批）。

## 8. 入库管理

### 8.1 流程描述

1. 基线落地跟踪完成后，自动生成入库审批
2. 基线库负责人进行审批（在系统角色中设置一人即可）
3. 审批通过：
   - 通过字段映射重构成基线库的结构（具体映射规则暂不明确）
   - 记录入库时间、来源项目、落地责任人
4. 审批不通过：
   - 记录来源项目、责任人、拒绝原因
   - **不允许修改后重新申请入库**

### 8.2 入库后数据管理

- 入库后的数据**允许修改**，修改后**需要重新审批**
- 基线库数据**需要版本管理**：每条记录记录变动内容即可

### 10.2 AI模型调用方式

- **尽量实时调用**，但不能卡死用户操作
- **长时间任务采取异步处理**，需提供进度反馈
- **部署方式**：调用外部API

### 10.3 AI生成结果管理

- AI生成结果的**采纳率需要统计**
- AI模型**不需要**支持用户反馈机制（除采纳/拒绝外）
- 临时表管理：每个分析任务对应一份，每次AI生成时删除前一次内容
- 支持"重新生成"按钮

### 10.4 预评审模型（模型13）——一期不实现

#### 10.4.1 目标

1. 补全人工未识别的风险（特别是界面/接口/交互类、环境耦合类）
2. 检验FMEA分析的系统性完整性（结构-功能-失效-措施是否闭环）
3. 针对变更点，预测涟漪影响（传播到非直接关联的部件/功能）

#### 10.4.2 四大引擎

| 引擎 | 原理 | 输出 |
|------|------|------|
| 引擎1：变更涟漪分析 | GNN学习历史变更传播模式，构建产品结构图+功能依赖图 | 遗漏风险（如远程部件影响预测） |
| 引擎2：结构-功能闭环检验 | NLP提取功能描述中的隐含涉及部件，与结构框图对比 | 功能孤岛、多余结构检测 |
| 引擎3：接口与交互风险挖掘 | 知识图谱关联规则挖掘，检索同类接口高发失效模式 | 接口失效遗漏检测 |
| 引擎4：系统性模式识别 | 图模式匹配与序列模式挖掘，提取失效链 | 跨层级FMEA条目补充建议 |

#### 10.4.3 不兼容问题检测

| 子模块 | 检测规则 |
|--------|----------|
| 结构-功能分配一致性 | 功能悬空、多余功能、功能滞后 |
| 失效传播链完整性 | 失效断链、影响遗漏、覆盖不全 |
| 措施兼容性与匹配性 | 措施冲突（红色）、措施脱节、重复浪费（黄色） |
| 变更涟漪跨层级验证 | 变更影响的系统级功能/失效是否在部件级DRBFM中覆盖 |

#### 10.4.4 关键设计原则

- **人机协作**：模型不自动写入FMEA，仅作为"第二双眼睛"提出建议，保留专家最终裁决权
- **增量学习**：专家"拒绝"或"采纳"模型建议作为训练样本，定期微调模型

#### 10.4.5 技术实现路径

| 阶段 | 方案 |
|------|------|
| 初期 | 规则为主：图数据库（Neo4j）+ Cypher查询实现检测规则 |
| 中期 | 混合：Sentence-BERT计算语义冲突分数 |
| 成熟期 | 图谱推理：跨层级知识图谱 + GNN预测 |

|----------|----------|----------|--------|

  - 一键导入1.0版本FMEA发布文件及审批信息到PMS（可行性待确认）

### 16.2 核心实体关系

```
Project 1──1 EvaluationTask(system) 1──1 AnalysisTask(system) 1──1 Baseline 1──1 LandingTask
        └──1 EvaluationTask(part)     └──1 AnalysisTask(part)

EvaluationTask 1──N EvaluationItem 1──N EvaluationDimensionOption
             1──N HistoryIssue

AnalysisTask 1──1 StructureDiagram 1──N StructureNode (无限层级树)
                                  1──N StructureEdge
                                  1──N InterfaceTable
            1──1 FunctionMatrix 1──N FunctionItem
                               1──N FunctionMapping
            1──1 FailureAnalysis 1──N FailureMode 1──N FailureCause
            │                                          1──N FailureEffect
            │                   1──N PreventiveMeasure
            │                   1──N DetectionMeasure
            │                   1──N OptimizationMeasure
            │                   1──1 SODRating
            1──N AIGenerationTemp
            1──1 Review 1──N Reviewer 1──N ReviewOpinion
                     1──1 MeetingMinutes
                     1──N ReviewNotification
            1──1 Approval

Baseline 1──N BaselineItem
LandingTask 1──N LandingItem 1──N LandingAudit

BaselineLibrary 1──N FailureLibrary 1──N MeasureLibrary
               1──N BaselineLibraryVersion
```
