## 1. 文档概述

### 1.1 文档目的

本文档为 FMEA 2.0 / DRBFM 全生命周期管理平台的功能规格文档，旨在：

1. 明确系统各功能模块的详细功能描述、操作流程及验收标准，为开发、测试和验收提供可量化、可验证的依据。
2. 将《FMEA2.0-需求分析文档 V2.0》中的业务需求转化为具体的功能规格，确保需求到实现的完整追溯。
3. 作为开发团队编码实现、测试团队编写测试用例、产品团队验收交付的统一基准。

### 1.3 术语定义

| 术语 | 全称 | 定义 |
|------|------|------|
| FMEA | Failure Mode and Effects Analysis | 失效模式与影响分析，一种系统化的可靠性分析方法 |
| DRBFM | Design Review Based on Failure Mode | 基于失效模式的设计评审，聚焦变更点的FMEA方法 |
| TSE | Test and Verification Engineer | 测试与验证工程师，系统级DRBFM的主要负责人 |
| PSE | Product System Engineer | 产品系统工程师，系统层面审核人 |
| PDTL | Product Development Team Lead | 产品开发团队负责人 |
| BG | Business Group | 业务集团（如CBG、SBG、EBG等） |
| TR | Technical Review | 技术评审节点（TR1~TR6） |
| PMS | Project Management System | 项目管理系统，FMEA 2.0的外部数据源系统 |
| CAS | Central Authentication Service | 中央认证服务，单点登录协议 |
| SOD | Severity / Occurrence / Detection | 严重度/发生度/探测度，FMEA风险评级三要素 |
| AP | Action Priority | 行动优先级，根据SOD组合判定（H/M/L） |
| I/M/N | Innovative / Modified / New | 开发类型选项：创新/修改/新增 |
| LEVEL | 层级编号 | 变更点清单中的层级深度，LEVEL=0表示系统级 |

### 1.5 验收标准编写规范

本文档中每个功能点的验收标准采用 AC（Acceptance Criteria）编号方式，遵循以下原则：

1. **具体性**：每条AC描述一个明确的、可观察的系统行为或输出结果。
2. **可测试性**：每条AC均可通过人工操作或自动化测试验证通过/失败。
3. **可追溯性**：每条AC可追溯到需求分析文档中的业务需求条目。
4. **原子性**：每条AC仅描述一个验证点，不混合多个验证逻辑。

## 2. 项目管理

### 2.1 项目列表

#### 功能描述

展示用户有权限访问的项目列表，支持按BG、状态、TR阶段进行筛选，帮助用户快速定位目标项目。

#### 前置条件

- 用户已通过CAS单点登录认证
- 用户的项目权限数据已从PMS系统同步至本地

#### 操作流程

1. 用户登录系统后，进入项目列表页面
2. 系统加载用户有阅读权限的项目列表
3. 用户可通过筛选条件（BG、TR阶段）缩小范围
4. 用户点击项目名称进入项目详情页

#### 验收标准

- **AC1**: 列表展示以下字段：项目名称、BG、PDTL、TR阶段、是否1级项目、创建时间，各字段数据完整显示
- **AC2**: 支持按BG下拉筛选（选项从已有项目数据中提取）、按TR阶段下拉筛选（TR1/TR3/TR4/TR4a/TR5/TR6），支持组合筛选
- **AC3**: 列表分页展示，每页默认20条，支持翻页和跳转指定页码
- **AC4**: 仅展示当前用户拥有阅读及以上权限的项目，无权限项目不可见
- **AC5**: 1级项目标识（is_level1字段）从PMS系统获取，缓存至Redis，缓存有效期30分钟；缓存过期后下次访问时重新从PMS拉取

---

### 2.2 项目创建

#### 功能描述

从PMS系统拉取项目信息，在本地创建项目记录，实现项目在FMEA 2.0平台中的初始化。

#### 前置条件

- 用户已登录且拥有项目管理权限
- 待创建项目在PMS系统中已存在

#### 操作流程

1. 用户在项目列表页点击"新建项目"按钮
2. 系统弹出创建对话框，用户输入PMS项目ID
3. 系统调用PMS数据源查询接口，自动拉取项目名称、BG、PDTL、TR阶段等信息
4. 用户确认信息无误后点击"确认创建"
5. 系统创建本地项目记录，并同步PMS项目成员权限

#### 验收标准

- **AC1**: 输入PMS项目ID后，系统自动从PMS数据源拉取并回填项目名称、BG、PDTL、TR阶段字段，用户无需手动填写
- **AC2**: 同一PMS项目ID不可重复创建，若输入已存在的PMS项目ID，系统提示"该项目已存在"并阻止创建
- **AC3**: 项目创建成功后，系统自动调用PMS权限同步接口，将PMS中该项目的成员及角色权限同步至本地`fmea_permission`表
- **AC4**: 1级项目标识由系统自动从PMS判定（调用`PmsQueryService.isLevel1Project()`），无需用户手动选择，判定结果存入`fmea_project.is_level1`字段

---

### 2.3 项目详情

#### 功能描述

展示项目基本信息、DRBFM评估任务和分析任务的进展状态，以及当前项目的权限列表，为用户提供项目全貌视图。

#### 前置条件

- 项目已创建
- 用户对该项目有阅读及以上权限

#### 操作流程

1. 用户在项目列表中点击项目名称
2. 系统进入项目详情页，展示项目基本信息
3. 页面展示系统级/部件级DRBFM任务进展
4. 页面展示项目权限列表

#### 验收标准

- **AC1**: 项目基本信息区域展示项目名称、BG、PDTL、TR阶段，数据与`fmea_project`表一致
- **AC2**: DRBFM任务进展区域分别展示系统级和部件级评估/分析任务的当前状态（如：草稿/已提交/审核通过/已评分/已确认/结构分析/功能分析/失效分析/基线输出/评审中/审批中/已完成），状态与任务状态机一致
- **AC3**: 权限列表区域展示当前项目所有成员的用户名、权限类型（管理/编辑/阅读/评审/审批），数据来源于`fmea_permission`表

### 2.4 权限管理

#### 功能描述

项目级权限管理，权限数据主要从PMS系统同步，管理权限用户可给其他人员开通编辑/阅读/评审/审批权限。

#### 前置条件

- 用户已登录
- 用户对目标项目拥有管理权限

#### 操作流程

1. 用户进入项目详情页的权限管理区域
2. 系统展示当前项目权限列表（从PMS同步+本地管理权限分配）
3. 管理权限用户点击"添加权限"按钮
4. 选择用户、选择权限类型（编辑/阅读/评审/审批），确认分配
5. 系统写入`fmea_permission`表

#### 验收标准

- **AC1**: 权限类型包括五种：管理、编辑、阅读、评审、审批，各权限类型的操作范围如下：
  - 管理权限：可给其他人开通编辑/阅读/评审/审批权限；拥有FMEA表单导出权限
  - 编辑权限：可编辑FMEA表单内容；拥有FMEA表单导出权限
  - 阅读权限：可查看项目FMEA内容
  - 评审权限：可参与DRBFM评审
  - 审批权限：可对DRBFM进行审批决策
- **AC2**: 权限数据从PMS系统同步至本地，本地不提供权限申请功能（用户需在PMS系统中申请项目权限）
- **AC3**: 拥有管理权限的用户可给其他人员开通编辑、阅读、评审、审批权限，开通后立即生效，写入`fmea_permission`表
- **AC4**: 管理权限不包含删除项目的权限，任何用户均无法在系统中删除已创建的项目

## 3. DRBFM触发评估

### 3.1 系统级评估任务创建

#### 功能描述

在项目下创建系统级DRBFM评估任务，作为DRBFM触发评估流程的起点。

#### 前置条件

- 项目已创建
- 当前用户为项目TSE角色
- 该项目尚无系统级评估任务

#### 操作流程

1. TSE进入项目详情页
2. 点击"新建系统级评估任务"按钮
3. 系统创建系统级评估任务，初始状态为"草稿"
4. TSE进入评估任务编辑页面

#### 验收标准

- **AC1**: 一个项目只能创建一个系统级评估任务（`level='system'`），重复创建时系统提示"该项目已存在系统级评估任务"并阻止创建
- **AC2**: 任务创建后状态为"草稿"（`status='draft'`），`submit_count`初始为0，`is_withdrawn`初始为0
- **AC3**: 任务创建人（`creator`字段）自动记录为当前登录用户，该用户默认为TSE角色

### 3.2 变更点清单导入

#### 功能描述

TSE通过上传Excel标准模板文件，将变更点清单导入系统，形成线上变更点清单数据。

#### 前置条件

- 系统级评估任务已创建
- 当前用户为TSE角色
- 已准备符合标准模板的Excel文件

#### 操作流程

1. TSE进入评估任务编辑页面
2. 点击"导入变更点清单"按钮
3. 选择Excel文件并上传
4. 系统解析Excel文件，写入`fmea_change_list`和`fmea_change_list_item`表
5. 导入完成后线上表格自动更新展示

#### 验收标准

- **AC1**: 支持Excel标准模板导入（.xlsx格式），模板包含以下列：LEVEL、ID、系统/部件名称、开发类型、I/M/N选项、变更点
- **AC2**: 导入字段完整映射：LEVEL→`level_num`，ID→`item_id`，系统/部件名称→`structure_name`，开发类型→`dev_type`，I/M/N选项→`imn_option`，变更点→`change_point`
- **AC3**: 支持无限层级结构，通过LEVEL列的数值判断层级深度，`parent_id`根据层级关系自动计算（LEVEL=n的父节点为最近的LEVEL=n-1的节点）
- **AC4**: 重复导入时覆盖线上已有变更点清单：先删除该评估任务关联的所有`fmea_change_list_item`记录，再写入新数据
- **AC5**: 导入完成后线上表格（EasyUI DataGrid）自动刷新，展示最新导入的变更点清单内容

### 3.3 质量策划表导入

#### 功能描述

TSE通过上传Excel文件导入质量策划表，系统读取各Sheet内容转Markdown后由AI处理，暂存质量策划内容。

#### 前置条件

- 系统级评估任务已创建
- 当前用户为TSE角色
- 已准备质量策划表Excel文件

#### 操作流程

1. TSE进入评估任务编辑页面
2. 点击"导入质量策划表"按钮
3. 选择Excel文件并上传
4. 系统读取Excel各Sheet内容，转Markdown格式
5. 调用AI服务处理Markdown内容，提取"风险小类+风险策划指南"字段组合
6. 质量策划内容暂存至`fmea_quality_plan`表

#### 验收标准

- **AC1**: 支持Excel文件导入（.xlsx格式），系统读取文件中所有Sheet的内容
- **AC2**: 系统将各Sheet内容转为Markdown格式，调用AI服务处理，提取"风险小类+风险策划指南"字段组合，存储至`fmea_quality_plan.raw_content`字段（JSON格式）
- **AC3**: 导入后质量策划内容暂存，不立即与变更点清单匹配，等待用户点击"自动匹配"或下次导入变更点清单时触发匹配

### 3.4 自动匹配

#### 功能描述

用户点击"自动匹配"按钮，系统将质量策划内容与线上变更点清单进行匹配，更新变更点描述字段。

#### 前置条件

- 系统级评估任务已创建
- 变更点清单和质量策划表至少已导入一项
- 当前用户为TSE角色

#### 操作流程

1. TSE在评估任务编辑页面点击"自动匹配"按钮
2. 系统读取线上变更点清单和质量策划暂存内容
3. 根据变更点清单中的部件名称与质量策划内容进行模糊匹配
4. 匹配成功后更新变更点描述字段
5. 展示匹配结果详情

#### 验收标准

- **AC1**: 系统读取线上`fmea_change_list_item`表中的变更点清单和`fmea_quality_plan`表中的质量策划内容进行匹配
- **AC2**: 匹配规则：根据变更点清单中的`structure_name`（部件名称）与质量策划中的风险小类进行模糊匹配（包含关系或关键词匹配）
- **AC3**: 匹配成功后，将质量策划中的"风险策划指南"内容更新至`fmea_change_list_item.quality_match`字段
- **AC4**: 匹配完成后，页面展示匹配详情弹窗，列出每条变更点的匹配结果（已匹配/未匹配），用户可查看具体匹配的质量策划内容

---

### 3.5 导入顺序处理

#### 功能描述

处理变更点清单和质量策划表的不同导入顺序，确保无论导入顺序如何，最终数据均能正确匹配。

#### 前置条件

- 系统级评估任务已创建
- 当前用户为TSE角色

#### 操作流程

**场景一：先导入变更清单再导入质量策划**
1. TSE导入变更点清单→变更清单直接更新线上表格
2. TSE导入质量策划表→质量策划暂存
3. 系统自动读取线上变更点清单，与质量策划内容匹配更新

**场景二：先导入质量策划再导入变更清单**
1. TSE导入质量策划表→质量策划暂存，不触发匹配
2. TSE导入变更点清单→变更清单更新线上表格
3. 系统自动读取质量策划暂存内容，与线上变更点清单匹配更新

**场景三：点击自动匹配**
1. TSE点击"自动匹配"按钮
2. 系统读取线上表单内容和质量策划暂存内容进行匹配

#### 验收标准

- **AC1**: 先导入变更清单再导入质量策划时，变更清单直接写入`fmea_change_list_item`表并更新线上表格；质量策划导入后系统自动读取线上变更点清单进行匹配，将匹配结果更新至`quality_match`字段
- **AC2**: 先导入质量策划再导入变更清单时，质量策划导入后仅暂存至`fmea_quality_plan`表，不触发匹配；变更清单导入后系统自动读取`fmea_quality_plan`暂存内容与变更点清单匹配更新
- **AC3**: 点击"自动匹配"按钮时，系统读取线上`fmea_change_list_item`表和`fmea_quality_plan`表的内容，执行匹配逻辑并更新`quality_match`字段

### 3.6 评估表提交与审核

#### 功能描述

TSE完成变更点清单编辑后提交评估表，PSE进行审核。审核通过后任务转回TSE进入历史问题导入步骤；审核不通过则通知TSE修改后重新提交。

#### 前置条件

- 系统级评估任务已创建
- 变更点清单已导入
- 当前用户为TSE角色（提交）或PSE角色（审核）

#### 操作流程

1. TSE确认变更点清单内容无误后，点击"提交审核"按钮
2. 系统将任务状态更新为"已提交"，PSE收到飞书/邮件通知
3. PSE查看评估表内容，进行审核：
   - 审核通过：任务自动转回TSE，进入历史问题导入步骤
   - 审核不通过：PSE填写不通过原因，系统通知TSE
4. TSE收到不通过通知后可修改评估表并重新提交
5. TSE提交后如需修改，可点击"撤回"按钮撤回已提交的评估表

#### 验收标准

- **AC1**: TSE提交后任务状态变为"已提交"（`status='submitted'`），系统通过飞书消息和邮件通知PSE，通知内容包含项目名称、评估任务ID、提交人、提交时间
- **AC2**: PSE审核通过后，任务状态自动变为"审核通过"（`status='approved'`），任务自动转回TSE，TSE可进入历史问题导入步骤
- **AC3**: PSE审核不通过时，系统通过飞书消息和邮件通知TSE，通知内容包含不通过原因；TSE可修改评估表后重新提交，任务状态变为"草稿"后可再次提交
- **AC4**: 系统记录提交次数（`submit_count`字段），每次TSE提交时`submit_count`自增1
- **AC5**: TSE提交后、PSE审核前，TSE可点击"撤回"按钮撤回评估表，撤回后任务状态变为"草稿"（`status='draft'`，`is_withdrawn=1`），TSE可继续修改后重新提交

---

### 3.7 历史问题导入

#### 功能描述

PSE审核通过后，TSE通过上传Excel模板文件导入历史问题，历史问题与系统级评估任务关联。

#### 前置条件

- 评估表已通过PSE审核（`status='approved'`）
- 当前用户为TSE角色
- 已准备符合模板格式的历史问题Excel文件

#### 操作流程

1. TSE进入评估任务页面，系统展示"历史问题导入"区域
2. TSE点击"导入历史问题"按钮
3. 选择Excel文件并上传
4. 系统解析Excel文件，写入`fmea_history_issue`表
5. 导入完成后页面展示历史问题列表

#### 验收标准

- **AC1**: 支持Excel模板导入（.xlsx格式），模板包含以下列标题
- **AC2**: 导入字段完整映射：故障单号→`fault_order_no`，问题编号→`issue_no`，问题类型→`issue_type`，涉及领域→`domain`，产品型号→`product_model`，问题描述→`issue_desc`，详细说明→`detail_desc`，解决方案→`solution`
- **AC3**: 历史问题与系统级评估任务关联（`evaluation_task_id`字段指向当前评估任务ID），支持同一评估任务多次导入（追加模式，不覆盖已有历史问题）
- **AC4**: 部件级评估任务可读取其所属项目的系统级评估任务的历史问题数据，在部件级评估页面提供"查看系统级历史问题"入口

---

### 3.8 五维风险评估

#### 功能描述

对LEVEL=0的变更点进行五维风险评估，五个维度分别为：技术新颖性、影响范围、失效严重度、变更复杂度、历史问题。每个维度提供高/中/低三组选项供用户选择。

#### 前置条件

- 评估表已通过PSE审核
- 历史问题已导入（可选）
- 当前用户为TSE角色

#### 操作流程

1. TSE进入评估任务页面，系统筛选出LEVEL=0的变更点清单项
2. 对每条LEVEL=0的变更点，展示五维评估表单
3. TSE逐项填写五个维度的选项（每个维度的高/中/低组选项中选择"是/否/空白"）
4. TSE填写完成后点击"评分"按钮

#### 验收标准

- **AC1**: 五个评估维度为：技术新颖性（`tech_novelty`）、影响范围（`impact_scope`）、失效严重度（`severity`）、变更复杂度（`change_complexity`）、历史问题（`history_issue`），每个维度在`fmea_evaluation_dimension_option`表中有对应的选项记录
- **AC2**: 每个维度的选项分组如下：
  - 高组：3个选项，每个选项可选"是/否/空白"
  - 中组：2个选项，每个选项可选"是/否/空白"
  - 低组：2个选项，每个选项可选"是/否/空白"
- **AC3**: 维度分数计算规则：高组有任一选项选中"是"→取高分（对应该维度的满分值）；高组无"是"时，看中组是否有"是"→取中分；中组无"是"时，看低组是否有"是"→取低分；三组均无"是"→得0分
- **AC4**: 五维评分必须由用户手动填写（TSE逐项选择），系统不自动填充选项
- **AC5**: 用户填完五维选项后，点击"评分"按钮触发风险综合评分计算（详见3.9节），未点击评分按钮前不计算评分

---

### 3.9 风险综合评分

#### 功能描述

根据五维评估结果计算风险综合评分，并自动判定风险等级（H/M/L），填充"DRBFM触发评估建议"和"最终DRBFM触发结论"列。

#### 前置条件

- LEVEL=0的变更点已完成五维评估选项填写
- 用户已点击"评分"按钮

#### 操作流程

1. TSE完成五维选项填写后点击"评分"按钮
2. 系统根据五维评分和权重计算风险综合评分
3. 系统根据评分判定风险等级
4. 系统自动填充"DRBFM触发评估建议"列和"最终DRBFM触发结论"列
5. 页面展示评分结果

#### 验收标准

- **AC1**: 计算公式为：风险综合评分 = 技术新颖性×15% + 影响范围×30% + 失效严重度×30% + 变更复杂度×20% + 历史问题×5%，结果保留两位小数
- **AC2**: 风险综合评分≥8分→风险等级为H（必须进行DRBFM分析），`risk_level='H'`
- **AC3**: 风险综合评分≥5分且<8分→风险等级为M（建议进行DRBFM分析），`risk_level='M'`
- **AC4**: 风险综合评分<5分→风险等级为L（可以不进行DRBFM分析），`risk_level='L'`
- **AC5**: 系统自动填充"DRBFM触发评估建议"列（`drbfm_suggestion`字段），内容根据风险等级映射：H→"必须进行DRBFM分析"，M→"建议进行DRBFM分析"，L→"可以不进行DRBFM分析"；同时填充建议理由（`drbfm_suggestion_reason`字段），用户可选择两种模式之一：分数映射预定义理由 / AI根据评分动态生成
- **AC6**: 系统默认填充"最终DRBFM触发结论"列（`drbfm_conclusion`字段），默认值与风险等级一致（H/M/L），PSE和TSE复核时可手动修改

---

### 3.10 评估确认与生成分析任务

#### 功能描述

PSE和TSE复核评分结果后确认评估表，系统校验评估完整性，确认后自动创建DRBFM分析任务并初始化子任务。

#### 前置条件

- 所有LEVEL=0的变更点已完成五维评估和风险综合评分
- 当前用户为PSE或TSE角色

#### 操作流程

1. PSE和TSE在评估表页面复核各变更点的五维评分和风险等级
2. 确认无误后点击"确认评估"按钮
3. 系统校验所有LEVEL=0的评估项五维选项是否已全部填写
4. 校验通过后，系统自动创建DRBFM分析任务
5. 分析任务初始化多个子任务

#### 验收标准

- **AC1**: 确认前系统校验所有LEVEL=0的评估项（`fmea_evaluation_item`中关联的变更点清单项`level_num=0`的记录），五维选项是否已全部填写（`fmea_evaluation_dimension_option`中每个维度的选项`is_selected`不为NULL）
- **AC2**: 校验未通过时，系统弹出提示信息，列出未完成评估的行号和变更点名称，格式如"第3行-XX部件：技术新颖性维度未填写"
- **AC3**: 校验通过后，系统自动创建DRBFM分析任务（写入`fmea_analysis_task`表），`evaluation_task_id`关联当前评估任务，`status`初始为"结构分析"（`structure`），评估任务状态更新为"已确认"（`confirmed`）
- **AC4**: 分析任务初始化时，系统创建以下子任务记录：结构分析、功能分析、失效分析（含失效模式/失效原因/失效影响/预防措施/探测措施/优化措施/SOD评分），各子任务按状态机顺序推进

---

### 3.11 部件级评估任务

#### 功能描述

在系统级评估基础上创建部件级评估任务。TSE在上传变更点清单或质量策划的界面可直接指派多名协作人，协作人编辑各自负责的变更点，无需签核。

#### 前置条件

- 系统级评估任务已完成（`status='confirmed'`）
- DRBFM分析任务已创建
- 当前用户为TSE角色

#### 操作流程

1. TSE进入项目详情页，点击"新建部件级评估任务"
2. 系统创建部件级评估任务
3. TSE在上传变更点清单或质量策划的界面，同时指派多名协作人
4. TSE统一上传变更点清单和质量策划表
5. 协作人编辑各自负责的变更点
6. TSE提交评估表（无需协作人签核）
7. PSE审核

#### 验收标准

- **AC1**: TSE可在上传变更点清单或质量策划的界面直接指派多名协作人（单方面指派，无需被指派人确认接受），指派信息记录在`fmea_permission`表中，`permission_type='edit'`，关联到对应变更点
- **AC2**: 指派后协作人自动获得对应变更点的编辑权限，协作人仅可编辑被指派的变更点，不可编辑其他变更点
- **AC3**: 变更点清单和质量策划由TSE统一上传（TSE上传完整的变更点清单和质量策划表），协作人不可自行上传
- **AC4**: 协作人编辑的变更点按`item_id`合并：`item_id`已存在→更新`change_point`等字段；`item_id`不存在→新增记录
- **AC4说明**: AC4描述的是变更点清单数据的合并规则（change_point等描述性字段的更新）
- **AC5**: DRBFM评估表合并规则：保留系统级（LEVEL=0）的评估内容不变，更新部件级评估内容，按行`item_id`匹配——已有评估内容的保留，否则新增（双路排序算法）
- **AC5说明**: AC5描述的是DRBFM评估结果的合并规则（五维评分、风险等级等评估数据的保留）
- **AC6**: 部件级评估无需协作人签核，TSE可直接提交评估表→PSE审核，审核流程与系统级一致（审核通过→进入历史问题导入→五维评估→评分→确认）

## 4. DRBFM分析

DRBFM分析任务是评估任务确认后的核心分析环节，按照步骤严格流转：结构分析 → 功能分析 → 失效分析 → 基线输出 → 评审 → 审批 → 完成。每个步骤产生关键交付物，步骤间存在数据依赖关系。

---

### 4.1 分析任务概览

#### 功能描述

展示DRBFM分析任务的整体进展，包括各子任务的完成状态，引导用户按步骤推进分析工作。

#### 前置条件

- DRBFM分析任务已创建（评估确认后自动生成）
- 当前用户对项目有编辑及以上权限

#### 操作流程

1. 用户进入项目详情页，点击分析任务进入分析概览页
2. 系统展示分析任务各步骤的进展状态
3. 用户按顺序点击各步骤进入对应子任务页面

#### 验收标准

- **AC1**: 分析概览页展示以下步骤及当前状态：结构分析→功能分析→失效分析（含预防/探测控制与风险评级），各步骤按顺序推进，前一步骤未完成时后续步骤不可操作（灰色不可点击）
- **AC2**: 当前可操作的步骤高亮显示，已完成的步骤标记"已完成"，未开始的步骤标记"未开始"
- **AC3**: 分析任务状态与子任务完成状态联动：结构分析完成→任务状态变为"功能分析"；功能分析完成→任务状态变为"失效分析"；失效分析完成→任务状态变为"基线输出"

---

### 4.2 结构分析

#### 功能描述

绘制结构框图，创建飞书文档和画板，以iframe方式嵌入页面。支持三种框图生成方式，用户可补充接口表格，框图节点根据风险评分标记颜色，接口线型按4类标记颜色。

#### 前置条件

- DRBFM分析任务已创建
- 分析任务当前步骤为"结构分析"
- 飞书API可用，飞书文档和画板创建接口可正常调用
- 评估表数据完整，DRBFM触发评估表已确认，含结构组成和风险评分数据
- 当前用户为TSE或协作人角色

#### 操作流程

1. 用户进入分析任务详情页，当前步骤为"结构分析"
2. 系统自动调用飞书API创建文档和画板，将画板以iframe方式嵌入页面
3. 用户选择框图生成方式：
   - **方式A - 一键导入**：若项目之前绘制过结构框图，点击"一键导入"按钮，系统读取历史框图的节点和边关系数据，重新渲染到画板
   - **方式B - 从系统级读取**：部件级分析任务可读取当前项目系统级分析任务的最新结构框图，提取关联部件的结构节点和边，渲染到画板
   - **方式C - 从评估表生成**：从评估分析表读取节点组成成分（按层级关系），解读为树结构构造节点信息，以Mermaid语法生成初图，保存为画板格式
4. 用户在接口表格中补充接口信息（起点、终点、功能、功能类型）
5. 框图节点根据评估表中的风险评分自动标记颜色：红色=H、黄色=M、灰色=L
6. 接口线型按4类自动标记颜色：物理=黑色、物质=蓝色、能量=绿色、信号=紫色
7. 用户可点击"导出"按钮，将框图导出为图片或PDF
8. 用户点击"下一步"，系统自动从画板中提取节点和边的关系数据，写入`StructureNode`和`StructureEdge`表

#### 验收标准

- **AC1**: 进入结构分析步骤时自动创建飞书文档和画板，iframe嵌入页面；`StructureDiagram`记录中`lark_doc_id`和`lark_board_id`非空
- **AC2**: 支持三种生成方式：一键导入历史框图、从系统级读取（部件级）、从评估表生成，分别测试三种方式，验证画板均能正确渲染结构框图
- **AC3**: 从评估表生成时，按层级关系读取节点组成成分，以Mermaid语法生成初图；选择"从评估表生成"后，画板中显示的框图与评估表的结构层级一致
- **AC4**: 用户可补充接口表格（起点、终点、功能、功能类型），在接口表格中录入数据并保存，刷新页面后数据不丢失；`InterfaceTable`表中有对应记录
- **AC5**: 框图节点根据风险评分标记颜色（红=H、黄=M、灰=L），H级节点边框/背景为红色，M级为黄色，L级为灰色；颜色与评估表风险等级一致
- **AC6**: 接口线型按4类标记颜色：物理=黑色、物质=蓝色、能量=绿色、信号=紫色，各类型接口线颜色与规范一致；`StructureEdge`的`interface_type`字段与线颜色对应
- **AC7**: 支持导出为图片/PDF，点击导出按钮后，浏览器下载对应格式的文件，内容与画板一致
- **AC8**: 进入下一步时自动生成节点和边的关系数据，点击"下一步"后，`StructureNode`和`StructureEdge`表中有完整的节点和边记录，与画板内容一致
- **AC9**: 用户确认结构分析完成后，系统将分析任务状态更新为"功能分析"（`status='function'`）

---

### 4.3 功能分析

#### 功能描述

基于结构框图生成功能矩阵图，器件为横轴、功能为纵轴。支持导入历史问题、技术要求文档、需求分析文档，支持AI从附件中提取功能补充到矩阵。功能矩阵确认后自动生成DRBFM表单"项目/功能"列。

#### 前置条件

- 结构分析步骤已完成，节点和边的关系数据已写入`StructureNode`和`StructureEdge`表
- 分析任务当前步骤为"功能分析"
- 当前用户为TSE或协作人角色

#### 操作流程

1. 用户进入功能分析步骤
2. 系统读取结构图中的节点信息（到器件一级）和边信息，自动生成功能矩阵：
   - 横轴：器件（`StructureNode`中`level`为器件级的节点）
   - 纵轴：功能（从`StructureEdge`中提取的功能描述）
   - 有关联的单元标记"Y"
   - H/M风险的单元格标记颜色
   - 功能排列按丰度（关联接口数量）降序排列
3. 用户可导入历史问题（关联系统级评估任务的历史问题数据）
4. 用户可导入技术要求文档（Word格式，AI处理提取功能）
5. 用户可导入需求分析文档（支持多份）
6. 用户点击"AI补充"按钮，AI从已上传的附件中提取功能，按规范（动词+名称）填充到纵轴
7. 用户校核矩阵关系，可手动编辑关联关系（一键确认OK或逐个确认OK）
8. 用户点击"确认"，系统自动将功能矩阵的横轴与纵轴组合，生成DRBFM表单"项目/功能"列
9. 用户点击"下一步"，进入失效分析步骤

#### 验收标准

- **AC1**: 系统读取`fmea_structure_node`（到器件一级，即最末级节点）和`fmea_structure_edge`中的边信息，将器件（节点）作为横轴、功能（边）作为纵轴，形成功能矩阵图，有关联的单元标记"Y"，数据写入`fmea_function_matrix`、`fmea_function_item`和`fmea_function_mapping`表
- **AC2**: 功能矩阵中有H/M风险的单元标记颜色：H→红色，M→黄色
- **AC3**: 功能来源不仅从结构图的边信息提取，也支持AI从附件中提取功能，`fmea_function_item.source`字段区分来源：`edge`（边信息提取）或`ai_attachment`（AI从附件提取）
- **AC4**: 功能排列顺序按功能的丰度（关联的接口数量）降序排列，`fmea_function_item.abundance_order`字段记录排序值
- **AC5**: 支持上传以下附件类型：
  - 历史问题（关联到系统级评估任务，从`fmea_history_issue`读取）
  - 技术要求（Word文档，通过Apache Tika提取内容后由AI处理）
  - 需求分析文档（可上传多份，Word/PDF格式）
- **AC6**: 用户上传附件后点击"AI补充"按钮，系统调用AI服务从附件中提取功能，按功能描述规范（动词+名称）生成功能项，填充到功能矩阵纵轴
- **AC7**: 功能描述规范：功能描述格式为"动词+名称（量词或状语）"，技术要求描述格式为"XX性能+量词"，功能描述必须清楚，功能之间无重叠冲突，一个系统元素可以有多个功能
- **AC8**: 支持专业域配置历史功能库（`fmea_function_library`表），用户可从功能库中选择功能添加到功能矩阵
- **AC9**: 功能矩阵确认后，系统将横轴（器件节点）与纵轴（功能）组合生成DRBFM表单的"项目/功能"列，写入失效分析表
- **AC10**: 用户确认功能分析完成后，系统将分析任务状态更新为"失效分析"（`status='failure'`）

### 4.4 失效分析

#### 功能描述

基于功能矩阵生成失效模式、原因、影响、措施。失效分析、预防与探测控制合并在一个步骤页面。AI生成结果存入临时表，用户在页面选择采纳后复制到正式业务表。

#### 前置条件

- 功能分析步骤已完成，功能矩阵已确认，DRBFM表单"项目/功能"列已生成
- 分析任务当前步骤为"失效分析"
- AI模型服务可用，失效模式/原因/影响/措施/SOD评分模型可正常调用
- 当前用户为TSE或协作人角色

#### 操作流程

1. 用户进入失效分析步骤（页面同时展示失效分析、预防与探测控制两个区域）
2. 用户点击"AI生成"按钮：
   - 系统读取功能矩阵中与变更相关的功能
   - 删除该分析任务对应类型的旧临时数据（`AIGenerationTemp`表中`analysis_task_id`和`generation_type`匹配的记录）
   - 调用AI模型依次生成：失效模式 → 失效原因 → 失效影响 → 预防措施 → 探测措施
   - 生成结果存入`AIGenerationTemp`临时表
3. 用户在页面查看AI生成结果，逐条选择"采纳"或"忽略"：
   - 采纳：数据从临时表复制到正式业务表（`FailureMode`、`FailureCause`、`FailureEffect`、`PreventiveMeasure`、`DetectionMeasure`），临时表记录标记"已采纳"
   - 忽略：临时表记录保留但不复制
4. 失效模式按7种类型生成，每种类型对应`fmea_failure_mode.mode_type`枚举值：

   | 类型 | 枚举值 | 描述 |
   |------|--------|------|
   | 异常 | ABNORMAL | 功能以错误方式执行 |
   | 相逆 | REVERSE | 功能反向执行 |
   | 伴随 | ACCOMPANY | 产生非预期附加功能 |
   | 过量 | EXCESS | 功能超出预期范围 |
   | 减量 | REDUCE | 功能低于预期范围 |
   | 部分 | PARTIAL | 功能部分执行 |
   | 空白 | BLANK | 功能完全丧失 |

   AI生成失效模式时，需同时关注变更点（关联关键结构/接口）可能带来的典型失效模式，不同失效模式分行列出

5. 失效原因按6个维度展开，每个维度对应`fmea_failure_cause.cause_dimension`枚举值：

   | 维度 | 枚举值 | 说明 |
   |------|--------|------|
   | 固有设计缺陷 | DESIGN | 设计本身存在的问题 |
   | 制造/装配问题 | MANUFACTURE | 生产过程中引入的问题 |
   | 应力诱发失效 | STRESS | 外部应力导致的失效 |
   | 耗损与老化 | WEAR | 长期使用导致的退化 |
   | 交互性/系统性失效 | INTERACTION | 系统间交互导致的问题 |
   | 综合应力作用 | COMPOSITE | 多种应力综合作用 |

   失效原因需结合变更点具象化（`change_point_ref`字段关联变更点），控制方法成熟的失效原因不用展开，其他系统失效且自身无法强化/检测的不用分析

6. 失效影响逐层识别，`fmea_failure_effect.effect_level`枚举值：

   | 层级 | 枚举值 | 说明 |
   |------|--------|------|
   | 上级/系统级 | SYSTEM | 对上级系统的影响 |
   | 整机级 | MACHINE | 对整机的影响 |
   | 客户级 | CUSTOMER | 对客户的影响 |

   需具体描述影响什么功能、什么性能，多个影响列在一行中无需分行，不同失效影响如相同需保证描述一致

7. 预防措施写入`fmea_preventive_measure`表，按优先原则排列：设计固有可靠性（`measure_type='design'`）> 过程控制（`measure_type='process'`）> 状态监控（`measure_type='monitor'`），遵循以下规则：
   - 预防措施与性能要求相关
   - 描述必须清楚、完整，必要时引用参考文献/标准
   - 将设计的安全边际作为预防措施考虑
   - 针对某个失效原因的所有预防措施写在一行

8. 探测措施写入`fmea_detection_measure`表，遵循以下规则：
   - 探测措施三要素：探测阶段（`phase`）+ 探测对象（`target`）+ 探测方法（`method`）
   - 能够探测失效原因、失效模式的探测措施都应填写（优先考虑探测失效原因）
   - 如单个探测措施有效性不够，可结合不同对象/时机/方法的探测措施
   - 探测度反映探测措施逼近真实失效机理的程度

9. 优化措施写入`fmea_optimization_measure`表，与预防措施、探测措施为并列关系

10. 用户可点击"重新生成"按钮，系统删除旧临时数据后重新调用AI生成
11. 用户点击"下一步"，进入SOD评分步骤

#### 验收标准

- **AC1**: 点击AI生成，读取功能矩阵中变更相关功能，自动生成失效模式/原因/影响/措施；点击"AI生成"后，页面展示AI生成的失效分析结果；`AIGenerationTemp`表中有对应记录
- **AC2**: AI生成结果存入临时表，用户在页面选择采纳；AI生成后，正式业务表中无数据；用户点击"采纳"后，正式业务表中有对应记录
- **AC3**: 采纳后数据从临时表复制到正式业务表，临时表标记"已采纳"；`AIGenerationTemp`记录的`is_adopted`字段为`true`；正式业务表中有相同内容
- **AC4**: 失效模式7种类型：异常、相逆、伴随、过量、减量、部分、空白；AI生成的失效模式`mode_type`字段值在7种枚举范围内；页面按类型分组展示
- **AC5**: 失效原因6个维度：固有设计缺陷、制造/装配问题、应力诱发失效、耗损与老化、交互性/系统性失效、综合应力作用；AI生成的失效原因`cause_dimension`字段值在6种枚举范围内；页面按维度分组展示
- **AC6**: 失效影响逐层识别：上级/整机/客户级；每条失效影响记录包含3个层级的描述；`FailureEffect`的`effect_level`字段区分system/machine/customer
- **AC7**: 预防措施优先原则：设计固有可靠性>过程控制>状态监控；页面中预防措施按类型排序展示；`PreventiveMeasure`的`type`字段值正确
- **AC8**: 探测措施三要素：探测阶段+探测对象+探测方法；每条探测措施记录的`phase`、`target`、`method`字段均非空；页面展示完整三要素
- **AC9**: 优化措施与预防措施、探测措施为并列关系，写入`fmea_optimization_measure`表
- **AC10**: 支持"重新生成"按钮，AI生成结果可多次重新生成；点击"重新生成"后，旧临时数据被删除，新结果生成；可反复操作
- **AC11**: 每次AI生成时删除该分析任务对应类型的旧临时数据；生成前后查询`AIGenerationTemp`表，确认旧记录已删除，仅保留最新生成结果
- **AC12**: 失效分析、预防与探测控制合并在一个步骤页面；页面同时展示两个区域，无需切换步骤即可完成全部操作

### 4.5 SOD评分

#### 功能描述

对失效模式进行严重度(S)、发生度(O)、探测度(D)评分。AI给出建议值，用户可自行修改。根据SOD组合查AP参考表判定优先级H/M/L。

#### 前置条件

- 失效分析数据已采纳，失效模式、原因、影响、措施数据已从临时表采纳到正式业务表
- AP参考表已配置，系统中存在完整的100行AP参考表数据
- SOD评价标准已配置

#### 操作流程

1. 失效分析数据采纳完成后，系统自动调用SOD评价模型，为每条失效模式生成S/O/D建议值
2. 页面展示AI建议值，用户可逐项修改：
   - **严重度(S)**：评价功能失效模式导致的最严重影响，取值1-10
   - **发生度(O)**：衡量预防措施有效性，模式1和模式2采用不同评分规则，取值1-10
   - **探测度(D)**：表示对失效原因或失效模式探测有效性的估计量度，取值1-10
3. SOD彼此独立评估，修改一个值不影响其他值
4. 用户确认SOD值后，系统根据SOD组合查询AP参考表，自动判定优先级H/M/L
5. 评分结果保存到`SODRating`表

#### 验收标准

- **AC1**: AI给出S/O/D建议值，用户可自行修改；页面展示AI建议值（`ai_suggested_s/o/d`），用户修改后保存的值与修改一致
- **AC2**: 严重度(S)评价功能失效模式导致的最严重影响；S值与失效影响的严重程度对应；参考SOD评价标准中严重度评分规则
- **AC3**: 发生度(O)衡量预防措施有效性，模式1和模式2采用不同评分规则；`SODRating`记录的`mode_type`字段正确标识模式1或模式2；两种模式的O值评分规则不同
- **AC4**: 探测度(D)表示对失效原因或失效模式探测有效性的估计量度；D值与探测措施有效性对应；D值越低表示探测越有效
- **AC5**: SOD彼此独立评估；修改S值后O和D值不变；修改O值后S和D值不变；修改D值后S和O值不变
- **AC6**: 根据SOD组合查AP参考表判定优先级H/M/L；给定S/O/D值后，系统查AP参考表得到的`ap_level`与标准表一致
- **AC7**: AP参考表为固定标准（100行），暂不允许修改；`APReference`表有且仅有100行数据；系统不提供AP参考表的编辑入口
- **AC8**: SOD评价标准适配：`fmea_sod_standard`表支持公司级默认规则（`scope='company'`）和BG级私有规则（`scope='bg'`），各BG可在默认规则基础上增改后保存为BG私有规则

### 4.6 步骤流转

#### 功能描述

分析任务按步骤严格顺序流转：结构分析 → 功能分析 → 失效分析 → 基线输出 → 评审 → 审批 → 完成。不可跳步，支持页面切换查看已完成步骤的内容。

#### 前置条件

- DRBFM分析任务已创建，`AnalysisTask`记录存在
- 当前步骤的数据已满足流转条件

#### 操作流程

1. 用户在当前步骤完成操作后，点击"下一步"按钮
2. 系统校验当前步骤的数据完整性：
   - 结构分析：`StructureNode`和`StructureEdge`表有数据
   - 功能分析：`FunctionMatrix`已确认，`FunctionItem`和`FunctionMapping`有数据
   - 失效分析：`FailureMode`、`FailureCause`、`FailureEffect`有数据，`SODRating`已填写
   - 基线输出：`BaselineItem`有数据，落地校验通过
   - 评审：评审意见全部闭环
   - 审批：审批已通过
3. 校验通过后，更新`AnalysisTask`的`status`为下一步骤
4. 用户可通过步骤导航栏切换查看已完成步骤的内容（只读模式，即基线模式）
5. 最终步骤"审批"通过后，任务状态变为"completed"

#### 验收标准

- **AC1**: 步骤严格按顺序流转，不可跳步；在结构分析未完成时点击功能分析步骤，系统拦截并提示"请先完成当前步骤"
- **AC2**: 每步完成后点击"下一步"进入下一环节；完成当前步骤操作后点击"下一步"，页面正确跳转到下一环节
- **AC3**: 支持页面切换查看已完成步骤的内容（基线模式）；点击已完成的步骤标签，页面展示该步骤内容且为只读模式
- **AC4**: 最终步骤"完成"后任务状态变为completed；审批通过后，`AnalysisTask`的`status`字段值为"completed"

### 4.7 AI辅助生成

#### 功能描述

在失效分析步骤中，用户可调用AI辅助生成失效模式、失效原因、失效影响、预防措施、探测措施和SOD评分建议。AI生成结果存入临时表，用户选择采纳后写入正式业务表。

#### 前置条件

- 功能分析已完成，功能矩阵数据已就绪
- 当前步骤为"失效分析"
- 当前用户为TSE或协作人角色

#### 操作流程

1. 用户在失效分析页面点击"AI生成"按钮
2. 系统读取功能矩阵中变更相关的功能
3. 系统调用AI服务，异步生成失效模式、失效原因、失效影响、预防措施、探测措施
4. 生成结果存入`fmea_ai_generation_temp`临时表
5. 前端轮询AI任务状态，完成后展示生成结果
6. 用户在页面上选择采纳/拒绝
7. 采纳后数据从临时表复制到正式业务表
8. 用户可点击"重新生成"按钮重新调用AI

#### 验收标准

- **AC1**: 用户点击"AI生成"按钮后，系统读取功能矩阵中与变更点相关的功能数据，调用`AiService`异步生成失效分析内容，返回`task_id`供前端轮询
- **AC2**: AI生成结果存入`fmea_ai_generation_temp`表，`generation_type`字段标识生成类型（failure_mode/failure_cause/failure_effect/preventive_measure/detection_measure），`is_adopted`初始为0
- **AC3**: 每次AI生成时，系统先删除该分析任务对应类型的旧临时数据（`DELETE FROM fmea_ai_generation_temp WHERE analysis_task_id=? AND generation_type=?`），再写入新生成的内容
- **AC4**: 前端通过jQuery.ajax轮询AI任务状态（`/ai/task/status?taskId=xxx`），间隔2秒，状态为COMPLETED时加载结果，FAILED时提示用户重试
- **AC5**: 用户在页面上选择采纳后，系统将临时表数据复制到对应的正式业务表（`fmea_failure_mode`/`fmea_failure_cause`等），临时表标记`is_adopted=1`
- **AC6**: 用户可点击"重新生成"按钮，系统删除前一次临时数据后重新调用AI生成，支持多次重新生成
- **AC7**: AI生成结果的采纳率需要统计，写入`fmea_ai_adoption_stat`表，记录`total_count`（总生成数）、`adopted_count`（采纳数）、`adoption_rate`（采纳率）
- **AC8**: AI调用超时设置：同步调用30秒超时，异步调用60秒超时；异步任务状态存储在Redis中，key格式`ai:task:{taskId}`，TTL 1小时

## 5. 基线输出与落地跟踪

基线输出是将失效分析结果转化为可执行措施清单的环节，落地跟踪则确保措施在项目中真正落地执行。审批通过后自动生成基线跟踪任务，落地负责人填报情况，主管审核闭环。

---

### 5.1 基线输出

#### 功能描述

根据失效分析结果输出基线清单，支持两种输出模式。措施标记来源和类型，每个措施需判断是否落地，落地的措施需指定负责人和时间。

#### 前置条件

- 失效分析步骤已完成，失效模式、原因、影响、措施、SOD评分数据完整
- 分析任务当前步骤为"基线输出"

#### 操作流程

1. 用户进入基线输出步骤
2. 系统默认按模式一生成基线清单，用户可在页面切换模式：
   - **模式一**：根据严重度×发生度列出预防措施、探测措施、优化措施（适合解决方案或软件）
   - **模式二**：直接列出所有预防措施、探测措施、优化措施（适合硬件/结构件）
3. 预防措施、探测措施、优化措施三者并列关系，均从失效分析结果中提取
4. 每条措施自动标记来源："AI生成"或"历史"
5. 每条措施自动标记类型："新增"或"优化"
6. 每个措施需判断是否落地（默认"是"），用户可改为"否"关闭落地
7. 选择"是"的措施需指定落地负责人和落地时间
8. 工程师可判断措施重复，进行去重（删除重复措施）
9. 用户点击"提交"，系统校验：
   - 所有"是否落地"为"是"的措施，落地负责人不能为空
   - 所有"是否落地"为"是"的措施，落地时间不能为空
   - 落地时间不能为过去时间
10. 校验通过后，基线清单数据保存到`Baseline`和`BaselineItem`表

#### 验收标准

- **AC1**: 支持两种输出模式，用户可在页面切换；页面提供模式切换按钮；切换后基线清单按对应模式重新生成
- **AC2**: 模式一：根据严重度×发生度列出预防措施、探测措施、优化措施；模式一下，仅展示S×O达到阈值的措施；清单按S×O降序排列
- **AC3**: 模式二：直接列出所有预防措施、探测措施、优化措施；模式二下，展示所有措施，不按S×O筛选
- **AC4**: 预防措施、探测措施、优化措施三者并列关系；页面中三种措施并列展示，不存在从属关系；`BaselineItem`的`measure_type`区分三种类型
- **AC5**: 措施标记来源："AI生成/历史"；每条措施的`source`字段为"AI"或"history"；页面展示来源标签
- **AC6**: 措施标记类型："新增/优化"；每条措施的`type`字段为"new"或"optimize"；页面展示类型标签
- **AC7**: 每个措施需判断是否落地（默认"是"），改为"否"则关闭落地；新增措施默认`is_landing`为`true`；改为"否"后，该措施不要求填写负责人和时间
- **AC8**: 选择"是"的措施需指定落地负责人和落地时间；`is_landing`为`true`的措施，`landing_owner`和`landing_date`字段必填
- **AC9**: 提交时校验：所有"是否落地"为"是"的措施，落地负责人和落地时间不能为空，时间不能为过去时间；提交时缺少负责人或时间，系统提示校验错误；落地时间为过去时间，系统提示"落地时间不能早于当前时间"
- **AC10**: 工程师可判断措施重复，进行去重（删除）；选择重复措施后点击"去重"，该措施从基线清单中删除；`BaselineItem`记录被物理删除

---

### 5.2 基线落地跟踪任务生成

#### 功能描述

审批通过后自动生成基线跟踪任务，包含所有"是否落地"为"是"的措施，每条措施生成一条落地项。

#### 前置条件

- 审批已通过，`Approval`记录的`conclusion`为"approved"
- 基线清单中存在"是否落地"为"是"的措施，`BaselineItem`中`is_landing`为`true`的记录存在

#### 操作流程

1. 审批通过后，系统自动创建`LandingTask`记录，关联`Baseline`
2. 系统遍历`BaselineItem`中`is_landing`为`true`的记录，为每条措施创建一条`LandingItem`
3. 每条落地项的`landing_status`初始化为"pending"（待填报）
4. 系统通过飞书消息+邮件通知各落地负责人

#### 验收标准

- **AC1**: 审批通过后自动生成基线跟踪任务；审批通过后，`LandingTask`表中有对应记录；`status`为"pending"
- **AC2**: 跟踪任务包含所有"是否落地"为"是"的措施；`LandingItem`记录数与`BaselineItem`中`is_landing=true`的记录数一致
- **AC3**: 每条措施生成一条落地项；每条`BaselineItem`（`is_landing=true`）对应一条`LandingItem`记录

---

### 5.3 落地情况填报

#### 功能描述

落地负责人填报落地情况，选项包括完成/试验失败/无法落地/延期。填报后提交个人主管审核。

#### 前置条件

- 基线跟踪任务已生成，`LandingTask`和`LandingItem`记录存在
- 当前用户为落地负责人，用户与`LandingItem`的`landing_owner`匹配

#### 操作流程

1. 落地负责人收到飞书/邮件通知，进入落地跟踪页面
2. 负责人选择落地情况：
   - **完成**：直接提交主管审核
   - **试验失败**：填写失败说明，提交主管审核
   - **无法落地**：填写无法落地原因，提交主管审核
   - **延期**：重新指定负责人和落地时间，提交主管审核
3. 提交后，`LandingItem`的`landing_status`更新为对应状态
4. 系统通知个人主管进行审核

#### 验收标准

- **AC1**: 落地情况选项：完成/试验失败/无法落地/延期；页面提供4个选项供选择；`LandingItem`的`landing_status`字段值在4种枚举范围内
- **AC2**: 填报后提交个人主管审核；提交后，`LandingAudit`表中有对应审核记录；主管收到通知
- **AC3**: "延期"需重新指定负责人和落地时间；选择"延期"后，页面要求填写新的负责人和落地时间；`LandingItem`的`new_owner`和`new_date`字段非空
- **AC4**: "试验失败/无法落地"需重新分析；选择"试验失败"或"无法落地"后，主管审核通过时系统触发重新分析流程

---

### 5.4 落地审核

#### 功能描述

主管审核落地情况，根据落地情况执行不同动作。审核不通过则通知负责人重新处理，无重试次数限制。

#### 前置条件

- 落地负责人已提交填报，`LandingItem`的`landing_status`已更新
- 当前用户为落地负责人的主管，主管身份通过组织架构确认

#### 操作流程

1. 主管收到飞书/邮件通知，进入落地审核页面
2. 主管查看落地情况及负责人填写的说明
3. 主管进行审核：
   - **审核通过**：
     - 落地情况为"完成"：回写`BaselineItem`的落地状态为"已完成"；将措施写入`MeasureLibrary`措施库，将失效数据写入`FailureLibrary`失效库
     - 落地情况为"延期"：更新`LandingItem`的负责人和落地时间为新指定的值，状态重置为"待填报"
     - 落地情况为"试验失败/无法落地"：触发重新分析流程，通知分析任务负责人
   - **审核不通过**：主管填写审核意见，通知落地负责人重新处理
4. 审核不通过无重试次数限制，负责人可多次重新填报

#### 验收标准

- **AC1**: 审核通过→按落地情况执行动作（完成→回写基线落地表+写入措施库和失效库；延期→更新负责人/时间；试验失败/无法落地→触发重新分析）；分别测试4种落地情况的审核通过流程，验证对应动作正确执行
- **AC2**: 审核不通过→通知负责人重新处理；主管审核不通过后，落地负责人收到通知；`LandingItem`状态重置为"待填报"
- **AC3**: 无重试次数限制；负责人可多次填报-审核不通过-重新填报，系统不限制次数

---

## 6. 评审与审批

评审与审批是DRBFM分析的质量保障环节。评审由指定角色的评审人参与，意见闭环后AI生成评审纪要；审批按变更等级对应不同层级审批人，简单审批流不支持多级会签。

---

### 6.1 发起评审

#### 功能描述

负责人指定评审角色与评审人，设置评审日期，发起评审。评审页面展示DRBFM分析内容及基线清单。

#### 前置条件

- 基线输出步骤已完成，基线清单已提交，校验通过
- 分析任务当前步骤为"评审"

#### 操作流程

1. 负责人进入评审步骤
2. 页面展示DRBFM分析内容（结构框图、功能矩阵、失效分析表）及基线清单（无论是否落地的措施都列出）
3. 负责人指定评审角色与评审人：
   - **必选角色**：专业组长、PSE、TSE、硬件质量经理、DFX SE（5个角色必须指定评审人）
   - **可选角色**：维护代表、制造代表、软件/固件工程师、射频/天线工程师、结构/热设计工程师、采购/供应商质量代表（6个角色可选）
   - 评审人在项目内动态配置，从项目成员中选择
4. 负责人设置评审日期
5. 负责人点击"发起评审"按钮
6. 系统创建`Review`记录，状态为"评审中"
7. 评审页面从编辑模式变为评审详情页面（只读展示+评审意见区域）
8. 系统向所有评审人发送飞书消息+邮件通知

#### 验收标准

- **AC1**: 评审页面展示DRBFM分析内容及基线清单（无论是否落地的措施都列出）；评审页面包含结构框图、功能矩阵、失效分析表、基线清单；基线清单中`is_landing`为"否"的措施也展示
- **AC2**: 负责人指定评审角色与评审人（项目内动态配置）；评审人选择器展示项目成员列表；选择后`Reviewer`记录中`role`和`user_id`正确
- **AC3**: 必选角色：专业组长、PSE、TSE、硬件质量经理、DFX SE；发起评审时，5个必选角色未指定评审人，系统提示"请指定必选角色评审人"
- **AC4**: 可选角色：维护代表、制造代表、软件/固件工程师、射频/天线工程师、结构/热设计工程师、采购/供应商质量代表；可选角色未指定评审人时不阻止发起评审
- **AC5**: 设置评审日期；评审日期字段必填；`Review`记录的`review_date`字段非空
- **AC6**: 发起后评审页面变为评审详情页面；发起评审后，页面从编辑模式切换为详情展示模式；评审人可查看内容并提交意见

---

### 6.2 评审通知

#### 功能描述

定时通知评审人，评审日期提前一周和提前一天各通知一次。到期未评审自动标记为"不通过"。

#### 前置条件

- 评审已发起，`Review`记录状态为"评审中"
- 评审日期已设置，`Review`的`review_date`非空

#### 操作流程

1. 评审发起后，系统注册两个定时通知任务
2. 评审日期提前7天，系统向所有未评审的评审人发送第一次通知（飞书消息+邮件），记录到`ReviewNotification`表
3. 评审日期提前1天，系统向所有未评审的评审人发送第二次通知（飞书消息+邮件），记录到`ReviewNotification`表
4. 评审日期到期时，系统检查所有评审人的评审状态：
   - 已提交意见的评审人保持原状态
   - 未提交意见的评审人，`Reviewer`记录的`conclusion`自动标记为"不通过"

#### 验收标准

- **AC1**: 评审日期提前一周通知一次；模拟时间到评审日期前7天，检查评审人收到飞书消息和邮件；`ReviewNotification`有`notify_type`="one_week"的记录
- **AC2**: 评审日期提前一天再通知一次；模拟时间到评审日期前1天，检查评审人收到飞书消息和邮件；`ReviewNotification`有`notify_type`="one_day"的记录
- **AC3**: 到期未评审则自动标记为"不通过"；评审日期到期后，未提交意见的评审人`Reviewer`记录`conclusion`为"rejected"
- **AC4**: 通知渠道：飞书消息+邮件；每次通知均通过飞书消息和邮件两个渠道发送

---

### 6.3 评审结论判定

#### 功能描述

根据评审人意见率判定评审结论。1级项目且意见率未达2/3时需组织线下会议。

#### 前置条件

- 评审已发起，评审人已收到通知
- 部分或全部评审人已提交意见

#### 操作流程

1. 系统从PMS获取项目是否为1级项目，更新`Review`的`is_level1_project`字段
2. 系统实时计算评审人意见率：已提交意见的评审人数 / 总评审人数
3. 判定逻辑：
   - 意见率≥2/3：进入意见闭环流程
   - 意见率<2/3且为1级项目：负责人需组织线下会议，上传会议纪要（文本格式），会议纪要关联到`MeetingMinutes`表
   - 意见率<2/3且非1级项目：仍可进入意见闭环流程
4. 会议纪要需关联到评审记录

#### 验收标准

- **AC1**: 判断项目是否1级项目（从PMS获取）；`Review`记录的`is_level1_project`字段值与PMS系统中的项目等级一致
- **AC2**: 计算评审人意见率是否达到2/3；意见率=已提交意见评审人数/总评审人数；页面实时展示当前意见率
- **AC2.5**: 所有评审人提交评审结论后（含到期自动不通过的评审人），系统自动计算意见率。意见率=已提交结论的评审人数/总评审人数
- **AC3**: 未达到2/3→负责人组织线下会议，上传会议纪要（文本）；1级项目意见率<2/3时，页面出现"上传会议纪要"入口；上传后`MeetingMinutes`记录存在
- **AC4**: 会议纪要需关联到评审记录；`MeetingMinutes`的`review_id`字段与`Review`记录的`id`一致
- **AC5**: 达到2/3→进行意见闭环流程；意见率≥2/3时，页面自动进入意见闭环环节

---

### 6.4 评审意见闭环

#### 功能描述

评审意见提出→负责人答复→提出人闭环。意见全部闭环后AI生成评审纪要。

#### 前置条件

- 评审结论已判定，意见率已达标或会议纪要已上传
- 评审人已提交意见，`ReviewOpinion`记录存在

#### 操作流程

1. 评审人提交评审意见，系统创建`ReviewOpinion`记录，`is_closed`为`false`
2. 负责人查看评审意见，填写答复内容
3. 系统通知意见提出人
4. 意见提出人确认闭环，`ReviewOpinion`的`is_closed`更新为`true`
5. 重复步骤1-4直到所有意见闭环
6. 所有意见闭环后，AI自动生成评审纪要（文本格式），保存到`Review`记录
7. 主页提供入口方便跟踪未闭环意见
8. 负责人确认评审纪要后，可发起审批

#### 验收标准

- **AC1**: 意见闭环流程：意见提出→负责人答复→提出人闭环；验证`ReviewOpinion`记录的完整生命周期：创建→答复→闭环
- **AC2**: 评审人不可委托他人代为评审；评审提交时校验当前用户是否为指定评审人；非指定评审人无法提交意见
- **AC3**: 意见闭环无时间限制；意见提出后，无论多久均可答复和闭环；系统不设置超时自动闭环
- **AC4**: 主页需要有入口方便跟踪未闭环意见；主页展示"未闭环评审意见"入口；点击后展示当前用户相关的未闭环意见列表
- **AC5**: 意见全部闭环后AI生成评审纪要（文本）；所有`ReviewOpinion`的`is_closed`为`true`后，AI自动生成评审纪要；`Review`记录中有纪要文本内容

---

### 6.5 发起审批

#### 功能描述

评审完成后负责人发起审批。页面展示DRBFM分析内容、基线清单、会议纪要。负责人选择审批人，按变更等级对应不同层级审批人。

#### 前置条件

- 评审已完成，评审意见全部闭环，评审纪要已生成
- 分析任务当前步骤为"审批"

#### 操作流程

1. 负责人进入审批步骤
2. 页面展示DRBFM分析内容、基线清单、评审会议纪要
3. 负责人根据变更等级选择审批人：
   - **初级变更**（Level 1）：专业组长审批——不影响接口的软件Bug修复、不涉及软件变更的B配套器件导入、文档勘误、不改变意图的布线优化
   - **中级变更**（Level 2）：部门经理审批——影响单一模块性能的硬件设计优化、软件功能模块版本升级、涉及软件变更的B配套器件导入、涉及重新认证的B配套器件导入
   - **重大变更**（Level 3）：研发总监审批——架构性与范围性变更、资源性与流程性变更、显著影响成本与产品路标变更
4. 界面提示对应层级，负责人选择具体审批人
5. 负责人点击"发起审批"按钮
6. 系统创建`Approval`记录，状态为"待审批"
7. 系统向审批人发送飞书消息+邮件通知

#### 验收标准

- **AC1**: 页面展示DRBFM分析内容、基线清单、会议纪要；审批发起页面包含三个区域，内容与评审步骤一致
- **AC2**: 负责人选择审批人（界面提示对应层级）；选择变更等级后，界面提示对应的审批层级和可选审批人
- **AC3**: 初级变更→专业组长审批；中级变更→部门经理审批；重大变更→研发总监审批；`Approval`记录的`level`字段与变更等级对应；`approver_id`为对应层级的审批人
- **AC4**: 简单审批流，不支持多级会签；审批记录中只有一个审批人；系统不提供添加多个审批人的入口

---

### 6.6 审批处理

#### 功能描述

审批人处理审批，查看DRBFM分析内容后给出审批意见和结论。审批通过生成基线跟踪任务，审批不通过可再次发起。

#### 前置条件

- 审批已发起，`Approval`记录状态为"待审批"
- 当前用户为审批人，用户与`Approval`的`approver_id`匹配

#### 操作流程

1. 审批人收到飞书/邮件通知，进入审批页面
2. 审批人查看DRBFM分析内容、基线清单、评审会议纪要
3. 审批人填写审批意见和审批结论（通过/不通过）
4. 审批通过：
   - `Approval`记录的`conclusion`更新为"approved"
   - 系统自动生成基线跟踪任务（参见5.2）
   - 分析任务状态更新为"completed"
5. 审批不通过：
   - `Approval`记录的`conclusion`更新为"rejected"
   - 系统通知发起人审批不通过及审批意见
   - 发起人查看审批意见历史，可修改后再次发起审批
   - 无次数限制

#### 验收标准

- **AC1**: 审批人收到飞书/邮件通知；审批发起后，审批人收到飞书消息和邮件
- **AC2**: 审批人查看DRBFM分析内容、基线清单、评审会议纪要；审批页面展示完整的三部分内容，与评审步骤一致
- **AC3**: 审批人填写审批意见和审批结论；`Approval`记录的`opinion`和`conclusion`字段非空
- **AC4**: 审批不通过→通知发起人，发起人可再次发起审批（无次数限制）；审批不通过后，发起人收到通知；可多次重新发起审批，系统不限制次数
- **AC5**: 审批通过→生成基线跟踪任务；审批通过后，`LandingTask`表中有对应记录；`AnalysisTask`状态为"completed"

## 7. 入库管理

入库管理是将基线落地跟踪完成后的措施数据沉淀到基线库的过程。入库需经过审批，审批通过后通过字段映射重构为基线库结构。入库后数据允许修改但需重新审批，且需版本管理。

---

### 7.1 入库申请

#### 功能描述

基线落地跟踪完成后自动生成入库审批，由基线库负责人进行审批。

#### 前置条件

- 基线落地跟踪已完成，所有落地项状态为"完成"或"无法落地"或"试验失败"（已触发重新分析的不参与入库）
- 存在可入库的措施数据，至少有一条落地状态为"完成"的措施

#### 操作流程

1. 基线落地跟踪任务中所有落地项完成闭环（完成/无法落地/试验失败均已处理）
2. 系统自动创建`ApprovalInbound`记录，状态为"待审批"
3. 系统通知基线库负责人（系统配置中指定的入库审批人）

#### 验收标准

- **AC1**: 基线落地跟踪完成后自动生成入库审批；落地跟踪任务全部闭环后，`ApprovalInbound`表中有对应记录；状态为"pending"
- **AC2**: 基线库负责人进行审批；入库审批通知仅发送给系统配置的基线库负责人；非基线库负责人无法操作审批

---

### 7.2 入库审批

#### 功能描述

基线库负责人审批入库申请。审批通过后通过字段映射重构为基线库结构，记录入库信息。审批不通过则记录拒绝原因，不允许修改后重新申请。

#### 前置条件

- 入库审批记录已生成，`ApprovalInbound`记录存在
- 当前用户为基线库负责人，用户身份与系统配置的基线库负责人匹配

#### 操作流程

1. 基线库负责人进入入库审批页面，查看措施数据详情
2. 负责人进行审批：
   - **审批通过**：
     - 系统通过字段映射将措施数据重构为基线库结构
     - 写入`BaselineLibrary`记录，包含入库时间、来源项目、落地责任人
     - 写入`FailureLibrary`记录（失效模式、原因、影响、严重度）
     - 写入`MeasureLibrary`记录（措施描述、类型、来源），关联到`FailureLibrary`
   - **审批不通过**：
     - 记录来源项目、责任人、拒绝原因到`ApprovalInbound`记录
     - 该基线数据不允许修改后重新申请入库

#### 验收标准

- **AC1**: 审批通过→通过字段映射重构成基线库结构；审批通过后，`BaselineLibrary`、`FailureLibrary`、`MeasureLibrary`表中有对应记录；字段映射正确
- **AC2**: 审批通过→记录入库时间、来源项目、落地责任人；`BaselineLibrary`记录的`entry_time`、`source_project`、`landing_owner`字段非空且正确
- **AC3**: 审批不通过→记录来源项目、责任人、拒绝原因；`ApprovalInbound`记录的`reject_reason`非空；`source_project`和责任人信息已记录
- **AC4**: 审批不通过→不允许修改后重新申请入库；审批不通过后，该基线数据的"申请入库"按钮不可点击或隐藏；`ApprovalInbound`记录标记为"rejected_final"

---

### 7.3 入库后数据管理

#### 功能描述

入库后的数据允许修改，修改后需要重新审批。基线库数据需要版本管理，每条记录记录变动内容。

#### 前置条件

- 数据已入库，`BaselineLibrary`记录存在且审批已通过
- 当前用户有基线库编辑权限，用户具有基线库的管理或编辑权限

#### 操作流程

1. 用户进入基线库管理页面，查看已入库数据
2. 用户修改数据（如措施描述、失效模式等）
3. 系统自动记录变动内容到`BaselineLibraryVersion`表：
   - 记录修改前的值和修改后的值
   - 记录修改人和修改时间
4. 用户提交修改后，系统创建重新审批记录
5. 基线库负责人审批：
   - 审批通过：版本号递增，修改内容生效
   - 审批不通过：修改不生效，数据恢复为修改前的值

#### 验收标准

- **AC1**: 入库后数据允许修改；已入库的数据字段可编辑；修改后可保存
- **AC2**: 修改后需要重新审批；修改保存后，数据状态变为"待审批"；审批通过前修改内容不对外生效
- **AC3**: 基线库数据需要版本管理：每条记录记录变动内容；`BaselineLibraryVersion`表中有对应记录；包含`version`、`change_content`、`changed_at`、`changed_by`字段；版本号递增

## 8. DRBFM迭代更新

### 8.1 迭代更新触发

#### 功能描述

PMS推送TR节点触发信息，触发DRBFM迭代更新。系统接收到PMS推送后自动启动对应项目及层级的DRBFM迭代更新流程。

#### 前置条件

1. 项目已在FMEA系统中创建且至少完成过一次DRBFM分析
2. PMS系统与FMEA系统的接口通道正常

#### 操作流程

1. PMS系统在TR节点到达前4周推送触发信息至FMEA系统
2. FMEA系统接收推送，解析触发信息中的项目编号、TR节点、层级（系统级/部件级）
3. 系统自动创建该TR节点对应的迭代版本
4. 系统通知项目PDTL及分析人员开始迭代更新

#### 验收标准

- **AC1**: PMS推送TR节点触发信息后，FMEA系统自动触发迭代更新流程，生成新的迭代版本记录
- **AC2**: 一期采用方案1：1.0版本发布后，每个TR节点4周前由PMS推送所有变更点信息，FMEA系统据此启动迭代
- **AC3**: 一期仅实现系统级和部件级各一次触发，不实现多次迭代触发逻辑

---

### 8.2 增量更新

#### 功能描述

迭代更新采用增量更新方式，保留历史版本的分析内容，仅对变更部分进行更新，变更来源从PMS系统获取。

#### 前置条件

1. 已存在上一版本的DRBFM分析数据
2. PMS已推送变更来源数据

#### 操作流程

1. 系统从PMS获取变更来源数据
2. 将变更点与上一版本的分析内容进行比对，标记新增/修改/删除项
3. 保留上一版本中未变更的分析内容，仅对变更相关部分开放编辑
4. 迭代更新完成后，需重新走完整的评审审批流程

#### 验收标准

- **AC1**: 更新方式为增量更新（非全量覆盖），仅变更相关部分被标记为待更新
- **AC2**: 之前版本的分析内容完整保留，未变更项在新版本中保持不变且不可编辑
- **AC3**: 变更来源从PMS系统获取，包括变更点清单、变更类型、变更描述等
- **AC4**: 迭代更新完成后，必须重新走完整的评审审批流程，不允许跳过或简化

---

### 8.3 版本管理

#### 功能描述

DRBFM分析版本与TR节点对应，支持版本历史查看和基线模式切换。

#### 前置条件

1. 项目已关联TR节点信息

#### 操作流程

1. 系统根据TR节点自动创建对应版本号
2. 版本发布后，该版本数据锁定为基线
3. 用户可在页面切换基线模式查看历史版本内容
4. 版本历史记录每次变更的具体内容

#### 验收标准

- **AC1**: 系统FMEA版本规则：TR1触发→TR3发布1.0→TR4迭代2.0→TR5迭代3.0，依此类推
- **AC2**: 部件级FMEA版本规则：TR3触发→TR4发布1.0→TR4a迭代2.0→TR5迭代3.0，依此类推
- **AC3**: 版本历史记录变更内容，包括变更点、变更时间、变更人、变更前后对比
- **AC4**: 支持页面切换基线模式查看历史版本，基线模式下数据只读不可编辑

## 9. AI模型集成

### 9.1 五维风险评估（模型2）

#### 功能描述

AI辅助五维风险评估规则生成和DRBFM建议生成。用户填完五维信息后，点击评分按钮触发AI计算，AI返回风险综合评分及DRBFM建议。

#### 前置条件

1. 用户已完成五维信息填写（新颖性、复杂性、接口依赖、失效历史、控制成熟度）
2. AI模型服务正常运行

#### 操作流程

1. 用户在五维风险评估页面填写五维信息
2. 点击"AI评分"按钮
3. 系统同步调用AI模型2，传入五维信息
4. AI计算风险综合评分（加权公式），返回评分结果
5. 根据用户选择的建议模式，生成DRBFM建议：
   - 模式A：分数映射预定义理由（根据评分区间映射到预定义建议文本）
   - 模式B：AI根据评分动态生成建议文本
6. 结果展示在页面上，用户可采纳或修改

#### 验收标准

- **AC1**: 用户填完五维信息后点击评分按钮触发AI计算，按钮点击后置为loading状态防止重复提交
- **AC2**: AI计算风险综合评分，采用加权公式：综合评分 = w1×新颖性 + w2×复杂性 + w3×接口依赖 + w4×失效历史 + w5×控制成熟度，权重可配置
- **AC3**: DRBFM建议支持两种模式：模式A（分数映射预定义理由）和模式B（AI根据评分动态生成），两种模式结果均可展示
- **AC4**: 用户在页面上通过下拉框或单选按钮选择建议模式，选择后AI按对应模式生成建议
- **AC5**: 同步调用，超时时间30秒，超时后返回错误提示"AI计算超时，请稍后重试"

### 9.2 功能分析（模型5）

AI辅助结构框图到功能转换、功能描述生成、从附件提取功能。AI从结构框图边信息和上传附件中提取功能，补充到功能矩阵纵轴。

3. AI从边信息中提取功能描述，从附件中提取功能描述
4. 前端轮询任务状态，任务完成后展示提取结果
5. 用户确认后，AI补充的功能填充到功能矩阵纵轴

#### 验收标准

- **AC1**: 从结构框图边信息提取功能，包括边的源节点、目标节点、连接类型、信号/能量/物料传递描述等

- **AC3**: AI补充功能填充到功能矩阵纵轴，与手动添加的功能并列展示，标记来源为"AI生成"
- **AC4**: 异步调用，超时60秒，超时后任务状态标记为失败并提示用户
- **AC5**: 前端通过jQuery.ajax轮询异步任务状态，轮询间隔3秒，任务完成/失败后停止轮询

### 9.3 失效模式生成（模型7）

#### 功能描述

AI基于功能偏差生成7类失效模式，同时关注变更点可能带来的典型失效模式。

#### 前置条件

1. 功能矩阵中已填写变更相关的功能
2. 变更点信息已从PMS获取

#### 操作流程

1. 用户在失效模式页面点击"AI生成失效模式"按钮
2. 系统异步调用AI模型7，传入功能矩阵中变更相关的功能及变更点信息
3. AI基于功能偏差生成7种类型失效模式：异常、相逆、伴随、过量、减量、部分、空白
4. 同时结合变更点生成典型失效模式
5. 前端轮询任务状态，完成后展示生成结果
6. 不同失效模式分行列出，用户可逐条采纳或修改

#### 验收标准

- **AC1**: 读取功能矩阵中变更相关的功能作为AI输入，仅包含标记为变更影响的功能
- **AC2**: 生成7种类型失效模式：异常（功能以错误方式执行）、相逆（功能反向执行）、伴随（产生非预期附加功能）、过量（功能超出预期范围）、减量（功能低于预期范围）、部分（功能部分执行）、空白（功能完全丧失）
- **AC3**: 同时关注变更点可能带来的典型失效模式，将变更点与功能偏差结合生成更具体的失效模式描述
- **AC4**: 不同失效模式分行列出，每行包含失效模式类型标签和具体描述
- **AC5**: 异步调用，超时60秒，超时后任务状态标记为失败并提示用户

### 9.4 失效原因生成（模型8）

#### 功能描述

AI从6个维度结合变更点生成失效原因，控制方法成熟的失效原因不展开。

#### 前置条件

1. 失效模式已确定
2. 变更点信息已从PMS获取

#### 操作流程

1. 用户在失效原因页面点击"AI生成失效原因"按钮

- **AC2**: 失效原因需结合变更点具象化，描述中包含具体的变更内容、涉及的零部件/接口/参数，而非泛化描述
- **AC3**: 控制方法成熟的失效原因不展开，即已有成熟预防/探测措施的失效原因仅列出维度标签，不生成详细描述
- **AC4**: 异步调用，超时60秒，超时后任务状态标记为失败并提示用户

### 9.5 失效影响生成（模型9）

### 9.7 探测措施生成（模型11）

#### 功能描述

AI生成探测控制措施，包含探测阶段、探测对象、探测方法三要素。

#### 前置条件

1. 失效原因和/或失效模式已确定

#### 操作流程

1. 用户在探测措施页面点击"AI生成探测措施"按钮
2. 系统异步调用AI模型11，传入失效原因及失效模式
3. AI生成探测措施，包含三要素：探测阶段+探测对象+探测方法
4. 优先探测失效原因，如单个措施有效性不够则组合不同对象/时机/方法
5. 前端轮询任务状态，完成后展示生成结果

#### 验收标准

- **AC1**: 探测措施包含三要素：探测阶段（如设计验证/生产检验/售后监测）+探测对象（如失效原因/失效模式/失效影响）+探测方法（如仿真/试验/检测），三要素完整
- **AC2**: 优先考虑探测失效原因，其次探测失效模式，最后探测失效影响
- **AC3**: 如单个探测措施有效性不够，可结合不同对象/时机/方法组合，例如"设计验证阶段探测失效原因（仿真分析）+生产检验阶段探测失效模式（功能测试）"
- **AC4**: 异步调用，超时60秒，超时后任务状态标记为失败并提示用户

### 9.8 SOD评分建议（模型12）

#### 功能描述

AI辅助SOD评分，给出严重度(S)、频度(O)、探测度(D)建议值，支持两种评分模式。

#### 前置条件

#### 操作流程

1. 用户在SOD评分页面点击"AI评分建议"按钮
2. 系统同步调用AI模型12，传入失效分析上下文及评分规则
3. AI根据评分规则给出S/O/D建议值
4. 用户可采纳建议值或自行修改
5. 系统根据S/O/D值查AP参考表得出AP等级

#### 验收标准

- **AC2**: 用户可自行修改AI建议值，修改后AP等级自动重新计算
- **AC3**: 模式1（解决方案/软件）和模式2（硬件/结构件）采用不同评分规则，AI根据项目类型自动选择对应规则

### 9.9 AI生成结果管理

#### 功能描述

管理AI生成结果的采纳与统计，包括临时存储、采纳复制、重新生成、采纳率统计。

#### 前置条件

1. AI模型调用完成并返回结果

#### 操作流程

1. AI生成结果存入临时表（fmea_ai_result_temp）
2. 用户在页面上查看AI生成结果，选择采纳或忽略
3. 采纳后，数据从临时表复制到正式业务表
4. 用户可点击"重新生成"按钮，系统删除该分析任务对应类型的旧临时数据后重新调用AI
5. 系统统计AI生成结果的采纳率

#### 验收标准

- **AC1**: AI生成结果存入临时表（fmea_ai_result_temp），字段包括：id、task_id（分析任务ID）、result_type（结果类型：失效模式/失效原因/失效影响/预防措施/探测措施/SOD评分）、result_content（JSON）、create_time、source_model（来源模型编号）
- **AC2**: 用户选择采纳后，数据从临时表复制到正式业务表（fmea_failure_mode/fmea_failure_cause/fmea_failure_effect/fmea_prevention/fmea_detection/fmea_sod），复制后临时表记录标记为"已采纳"
- **AC3**: 支持"重新生成"按钮，点击后删除该分析任务对应类型的旧临时数据，重新调用AI模型生成
- **AC4**: 每次AI生成时，先删除该分析任务（task_id）对应类型（result_type）的旧临时数据，再写入新数据
- **AC5**: 临时表没有有效期，每个分析任务对应一份临时数据，不自动清理
- **AC6**: 统计AI生成结果采纳率，公式：采纳率 = 已采纳条数 / 总生成条数 × 100%，按模型类型、项目、时间维度统计
- **AC7**: 前端通过jQuery.ajax轮询异步任务状态，轮询接口返回任务状态（PENDING/RUNNING/SUCCESS/FAILED）及进度百分比

## 13. 外部系统集成

### 13.1 PMS系统集成

#### 功能描述

与PMS系统的数据交互，包括TR节点触发、项目判定、权限信息、变更来源数据获取。

#### 前置条件

1. PMS系统接口可用
2. 网络连通性正常

#### 操作流程

1. PMS系统推送数据至FMEA系统
2. FMEA系统接收并处理数据
3. FMEA系统按需缓存PMS查询结果

#### 验收标准

- **AC1**: PMS推送TR节点触发信息，FMEA系统提供接收接口，接口参数包括：项目编号、TR节点、触发时间、层级
- **AC2**: 从PMS获取1级项目判定，调用PMS接口查询项目是否为1级项目，返回布尔值
- **AC3**: 从PMS获取项目权限信息，包括项目成员角色（PDTL/分析人员/评审人员等），权限数据同步至本地缓存
- **AC4**: 从PMS获取变更来源数据，包括变更点清单、变更描述、变更类型、变更时间
- **AC5**: PMS数据源为只读，FMEA系统对PMS数据库仅配置SELECT权限账号，禁止INSERT/UPDATE/DELETE操作
- **AC6**: PMS查询结果按需缓存至Redis，缓存Key格式：`pms:{dataType}:{projectId}`，缓存有效期24小时，过期后重新查询

---

### 13.2 飞书文档集成

#### 功能描述

结构框图画板创建，自动创建飞书文档和画板，iframe嵌入FTL页面。

#### 前置条件

1. 飞书开放平台应用已创建
2. 飞书API权限已授权

#### 操作流程

1. 用户在结构框图步骤点击"创建画板"按钮
2. 系统调用飞书API创建文档和画板
3. 获取画板URL存入数据库
4. FTL页面通过iframe嵌入画板
5. 用户在画板中编辑结构框图
6. 系统回读画板节点和边数据

#### 验收标准

- **AC1**: 自动创建飞书文档和画板，调用飞书开放平台API，创建成功后返回文档ID和画板ID
- **AC2**: 获取画板URL存入数据库（fmea_structure_diagram表board_url字段），与项目分析任务关联
- **AC3**: iframe嵌入FTL页面，iframe src指向画板URL，设置宽高100%自适应容器，支持画板内编辑操作
- **AC4**: 支持回读画板节点和边数据，调用飞书画板数据读取API，解析节点（结构元素）和边（连接关系）信息
- **AC5**: 支持导出画板为图片（PNG）/PDF，调用飞书画板导出API
- **AC6**: 复用PMS已有飞书接口（应用凭证、Token获取等），不单独创建飞书应用

- **AC2**: 支持发送卡片消息，卡片包含：标题、正文摘要、操作按钮（如"前往审批"），点击按钮跳转至FMEA系统对应页面
- **AC3**: 通知场景：审核通知（提交审核时通知审核人）、审批通知（提交审批时通知审批人）、评审通知（评审会安排时通知评审角色），每种场景的消息模板独立配置

### 13.4 CAS单点登录

#### 功能描述

对接CAS登录，用户访问FMEA系统自动跳转CAS认证。

#### 前置条件

1. CAS服务器已部署且可访问
2. FMEA系统已在CAS注册为客户端

#### 操作流程

1. 用户访问FMEA系统
2. 系统检测未登录，重定向至CAS登录页
3. 用户在CAS完成认证
4. CAS回调FMEA系统并携带认证票据
5. FMEA系统验证票据，建立本地会话
6. 同步用户基本信息至本地缓存

#### 验收标准

- **AC1**: 用户访问FMEA系统自动跳转CAS登录页，未认证用户302重定向至CAS登录URL
- **AC2**: CAS认证成功后回调FMEA系统，FMEA系统验证Service Ticket有效后创建本地Session
- **AC3**: 本地维护用户基本信息缓存（用户ID、姓名、邮箱、部门），缓存至Redis，有效期与Session一致
- **AC4**: 权限信息从PMS同步，不依赖CAS，CAS仅负责身份认证

## 14. 非功能性需求规格

### 14.1 性能需求

#### 功能描述

系统性能指标要求。

#### 验收标准

- **AC1**: 支持100人并发访问，系统在100并发用户下正常运行，无请求丢失
- **AC2**: 页面响应时间<3秒（常规操作，不含AI调用），常规操作包括：页面加载、数据查询、表单提交、文件导出
- **AC3**: AI异步任务不阻塞用户操作，用户发起AI生成请求后可继续操作其他功能，AI任务在后台执行

---

### 14.2 可用性需求

#### 功能描述

系统可用性指标要求。

#### 验收标准

- **AC1**: 系统可用性≥90%（按月统计），即每月故障时间不超过72小时
- **AC2**: 数据永久保存，业务数据不设置自动删除策略，数据库定期备份（每日全量备份+实时增量备份）

---

### 14.3 安全需求

#### 功能描述

系统安全防护要求。

#### 验收标准

- **AC1**: CAS单点登录认证，所有请求必须经过CAS认证，未认证请求重定向至CAS登录页
- **AC2**: 项目级RBAC授权，基于角色的访问控制，权限粒度到项目级别，不同角色（PDTL/分析人员/评审人员/管理员）具有不同操作权限
- **AC3**: HTTPS传输加密，所有HTTP请求强制跳转HTTPS，TLS版本≥1.2
- **AC4**: MyBatis Plus参数化查询防SQL注入，所有数据库查询使用MyBatis Plus的参数化绑定，禁止拼接SQL字符串
- **AC5**: FTL输出自动转义防XSS，FreeMarker模板默认启用HTML自动转义（`output_format=HTMLOutputFormat`），用户输入内容渲染时自动转义特殊字符
- **AC6**: Spring Security CSRF Token，所有POST/PUT/DELETE请求携带CSRF Token，Token验证失败返回403
- **AC7**: PMS数据源仅配置SELECT权限账号，数据库连接池中PMS数据源的账号仅授予SELECT权限，通过数据库层面限制写操作

---

### 14.4 审计日志

#### 功能描述

关键操作自动记录审计日志。

#### 验收标准

- **AC1**: 关键操作自动记录审计日志，关键操作包括：登录/登出、数据创建/修改/删除、审批/审核操作、配置变更、AI模型调用
- **AC2**: 日志内容：用户ID（user_id）、操作类型（action_type：CREATE/UPDATE/DELETE/APPROVE/REJECT/LOGIN/LOGOUT）、资源类型（resource_type：PROJECT/FMEA/BASELINE/CONFIG等）、资源ID（resource_id）、操作详情JSON（detail_json）、IP地址（ip_address）、时间戳（timestamp）
- **AC3**: 审计日志不可删除，仅可查询，数据库层面通过触发器或权限控制禁止DELETE操作，查询支持按用户ID、操作类型、时间范围筛选
